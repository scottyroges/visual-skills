1The root cause and the fix — one WHERE clause

The bug lived in a single query. syncLiveStandings keeps tournamentResult rows updated with projected numbers while a tournament is IN_PROGRESS — that's a feature for standings, but getSeasonStats aggregated those same rows with no status filter, so the golfer profile treated projections as history.

golfer.ts src/server/repositories/golfer.ts

+1

The one-line root-cause fix — getSeasonStats now aggregates only settled tournaments.

  • ›adds .where("tournament.status", "in", ["COMPLETED", "FINALIZING"]) to the stats query — the exact filter the earnings rollup already applies in repositories/earnings.ts, so stats and standings now agree on what counts
  • ›effect on each stat: totalEarnings drops projected money, bestFinish no longer reports a live leader's "1st", eventsPlayed no longer counts an unfinished event
  • ›FINALIZING stays included on purpose: play is over and projected earnings are held until official results land, so those numbers are near-final rather than mid-round noise
  • ›the sibling getSeasonResults query is deliberately untouched — the results table should still show the live tournament, just clearly labeled
  • ›coverage note: the existing service tests mock this repo, so the new filter rides on typecheck plus the PR test plan's next-live-tournament verification
