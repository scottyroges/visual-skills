TL;DR

Let players plan their whole season — and let a cron job, not the user, turn plans into real picks.

WhatA Season Planner: slot golfers into any future tournament as PLANNED prepicks; the daily cron promotes valid plans to LOCKED in the database when the tournament starts.
WhyPPGL's strategy game is season-long — allocate ~200 golfers across ~40 weeks, no reuse. But picks were week-at-a-time: no way to reserve Scheffler for the Masters in March, or see the whole board at once.
ClosesTurns the app from a weekly chore into a persistent strategic tool — the full remaining schedule with prepick slots, results behind you, plans ahead of you.
Size2 PRs · 5 new service methods · 5 new tRPC endpoints · 1 cron auto-lock routine · 1 page + picker flow. No schema migration — Pick already carries status + lockedAt.
The big idea
“A plan costs nothing until the cron makes it real.”

A PLANNED pick is a soft reservation — invisible to opponents, freely editable, and it does not consume the golfer. Only the unattended cron promotion to LOCKED burns the golfer, earns money, and shows the pick to the league. If a plan turns out invalid on start day, it goes stale instead of burning the golfer: the user gets a $0 week, never a lost asset.

How it works

two writers, one row

Two paths write the same pick row. The user path (top) creates and edits PLANNED rows through the planner page — cheap, reversible, private. The cron path (bottom) runs inside /api/cron/sync: when a tournament's start date passes, startTournaments flips it UPCOMING → IN_PROGRESS and — in the same transaction — lockPlannedPicksForTournament validates each plan (golfer in the published field, golfer not already used this season) and promotes the valid ones to LOCKED. Invalid plans are skipped, not burned. Everything downstream of LOCKED — earnings rollup, opponent visibility — is preserved machinery that never learns PLANNED exists.

The pieces

6 components

One new page, one new cron routine, and a widened pick service — all writing the same Pick row. The signature shape of the whole phase is the two lives of a pick:

■ PLANNED — soft reservation
Invisible to opponents, freely editable, and it does not consume the golfer. Change it, delete it, or plan the golfer nowhere else — no cost.
status: PLANNED · lockedAt: null
→
cron: validate(inField, notUsed) → promote
■ LOCKED — committed pick
Burns the golfer for the season, earns real prize money via the rollup, and is visible to the league. Identical whether locked manually or by cron.
status: LOCKED · lockedAt: startDate

Every pick row is exactly one of these two states, and @@unique([leagueMemberId, tournamentId]) guarantees one row per member per tournament — so the planner and the Make Pick page can never diverge.

Key decisions

8 locked · 3 contested

The whole design hangs on the first two: what a plan reserves, and who turns it into a real pick.

1
What does a PLANNED pick reserve?
Nothing — it's a soft reservation. Only LOCKED picks consume a golfer for the season.
Planning three months out shouldn't cost options today. The exhausted-golfer query counts only LOCKED picks, so a golfer can be planned for week 10 and picked for week 8.
RejectedHard reservation (PLANNED blocks the remaining pool) — one speculative plan would lock away a golfer the user may never actually field, and stale plans would silently shrink the pool.
2
How does a plan become a real pick?
Write-side: the daily cron promotes valid PLANNED → LOCKED in the database on tournament start day.
After promotion the pick is indistinguishable from a manually submitted one — it burns the golfer, earns via the rollup, and is visible to opponents. One code path downstream, forever.
RejectedExtending read-side locking (compute LOCKED at read time, never write). Earnings, opponent visibility, and pool queries would each need PLANNED-aware special cases — three subtle forks instead of one DB write.
3
When is field membership validated?
At lock time (cron), never at plan time.
Tournament fields aren't published until a few days before the event — plan-time validation is impossible for most of the season, so validation moves to the only moment the field is known.
RejectedPer-tournament "likely in field" predictions in the picker — there's no historical field-entry data to predict from. Deferred; the picker shows the global remaining pool instead.
4
What happens to a plan that fails validation on start day?
It goes stale — skipped by the cron and flagged "missed" on the timeline. The user gets $0 that week but keeps the golfer.
Burning a golfer the user never actually fielded would punish twice for one mistake. Worst case is a lost week, never a lost asset.
5
What if Make Pick submits over an existing PLANNED pick?
submit updates the existing pick's golfer in place; LOCKED picks are rejected with PRECONDITION_FAILED.
@@unique([leagueMemberId, tournamentId]) guarantees one row per member per tournament — both flows converge on that row instead of racing to create duplicates.
6
What does the planner's season query return?
Raw DB status from getAllSeasonPicks — no read-side masking.
The planner must distinguish PLANNED (editable), LOCKED (done), and stale PLANNED (missed). Read-side locking would erase exactly the distinction the page exists to show.
7
Where does the auto-lock code live?
A pick-lock module built on db-core — no server-only guard — following the earnings-core pattern.
The cron API route and tests both need to call it. Its two validation queries are inlined rather than spun into a new repo file — not worth the indirection.
8
How do users assign a golfer to a week?
Tap-to-assign: tap a tournament row, pick from a searchable, world-ranking-sorted drawer.
Simpler to build and works on mobile, where most picks happen.
RejectedDrag-and-drop planning — deferred to Phase 10 polish. High interaction cost on touch screens for zero strategic benefit in v1.

Risks

The risk profile is concentrated in one place: an unattended cron writing money-bearing rows.

⚠The cron misses or fails the start-day run — plans never lock, users silently get $0 weeks.
Mitigation: Catch-up by construction: startTournaments selects every UPCOMING tournament with startDate <= now on every run, so the next successful run promotes whatever the failed one missed. Promotion is idempotent — it only touches rows still PLANNED.
⚠Partial failure mid-promotion: tournament flipped to IN_PROGRESS but its picks left unlocked.
Mitigation: One transaction per tournament: the status flip and lockPlannedPicksForTournament share a withTransaction scope — either both commit or neither does.
⚠PLANNED rows leak into money or visibility paths and inflate standings or spoil picks.
Mitigation: Explicit LOCKED filters in both earnings rollup queries and the league current-week view, each covered by tests — PLANNED is invisible everywhere except the owner's planner.
⚠A user edits a plan at the same moment the cron locks it.
Mitigation: Start-date guard on every mutation: updatePlan/deletePlan re-check the tournament and reject with PRECONDITION_FAILED once it has started; the cron only touches rows still PLANNED. No cross-member race exists — golfer uniqueness is per-member.
⚠Stale plans confuse users — a week quietly earns nothing.
Mitigation: Visible, not deleted: stale plans stay on the timeline with a "missed" indicator so users see exactly what slipped, and can clear it themselves.

Auto-lock algorithm — the promotion loop, both checks, and the transactionpick-lock.ts · sync.ts

The promotion runs inside startTournaments (src/server/services/sync.ts), which wraps each starting tournament in one transaction:

const lockResult = await withTransaction(async (tx) => {
  await syncRepo.updateTournamentStatus(tournament.id, TournamentStatus.IN_PROGRESS, tx);
  return lockPlannedPicksForTournament(tournament.id, tournament.season, tx);
});

lockPlannedPicksForTournament (src/server/repositories/pick-lock.ts — built on db-core, no server-only guard, executor-injectable for transactions and tests) then walks every PLANNED pick for the tournament:

  1. Field check — the golfer must have a CONFIRMED tournamentField entry for this tournament. Not in the field → the plan is not promoted (skip mode returns the golfer to the pool; leagues can opt into burnStalePicks to lock anyway).
  2. Usage check — count the member's LOCKED picks for this golfer this season; at or above the league's maxGolferUses (default 1) → skipped, stays unpromoted.
  3. Promote — passes both → UPDATE pick SET status = 'LOCKED', lockedAt = now().
  4. Cleanup — after locking, any other PLANNED picks for golfers that just became exhausted are removed, so the planner never shows a plan that can no longer lock.

Returns { locked, skipped, cleaned }, which the cron logs per tournament. Each pick is processed independently — golfer uniqueness is per-member, so there is no cross-member race.

tRPC surface — five new endpoints on the pick router3 mutations · 2 queries

All endpoints are protectedProcedures scoped through resolveLeagueMemberId(userId, leagueId) — a user can only ever touch their own membership's picks.

Endpoint Kind Service Semantics
pick.plan mutation planPick Creates a PLANNED pick. Rejects: unknown tournament/golfer (NOT_FOUND), started tournament (PRECONDITION_FAILED), existing pick for the tournament (CONFLICT).
pick.updatePlan mutation updatePlannedPick Swaps the golfer on an existing PLANNED pick. NOT_FOUND unless the pick exists and is PLANNED — LOCKED picks are unreachable.
pick.deletePlan mutation removePlannedPick Deletes a PLANNED pick; same PLANNED-only guard, plus the tournament must not have started.
pick.plannedPicks query getPlannedPicks The member's PLANNED picks for the season.
pick.allSeasonPicks query getAllSeasonPicks Every pick (both statuses) with tournament, golfer, purse, and result data — the planner timeline's spine, ordered by start date.

The Make Pick flow's guard is the same shape: an existing PLANNED pick is updated in place; a LOCKED pick rejects with PRECONDITION_FAILED.

Pick schema — no migration neededprisma/schema.prisma

The phase rides entirely on the existing model — status defaults to PLANNED, lockedAt records the promotion moment, and the unique constraint is what makes update-in-place safe:

model Pick {
  id             String     @id @default(uuid())
  leagueMemberId String     @map("league_member_id")
  tournamentId   String     @map("tournament_id")
  golferId       String     @map("golfer_id")
  status         PickStatus @default(PLANNED)
  createdAt      DateTime   @default(now()) @map("created_at")
  updatedAt      DateTime   @updatedAt @map("updated_at")
  lockedAt       DateTime?  @map("locked_at")

  @@unique([leagueMemberId, tournamentId])
  @@map("pick")
}

enum PickStatus {
  PLANNED
  LOCKED
}

As shipped — how the live code evolved past this plan7 deltas, same model

The phase shipped as designed, and the PLANNED/LOCKED model then absorbed every later feature without a schema change — the best evidence the core decisions held. Verified deltas between this plan and today's code:

  • Cron cadence: daily 10:00 UTC → every 15 minutes (vercel.json), so locks land within minutes of a start time rather than hours.
  • Route: the planner shipped at /season (NavBar "Season") rather than /planner, merged with the season history view.
  • maxGolferUses: the binary "already used" check generalized to a per-league use limit (findExhaustedGolferIds), letting leagues allow golfer reuse.
  • Stale-plan handling hardened: skip-mode now deletes a not-in-field plan at lock time (golfer returns to the pool immediately), leagues can opt into burnStalePicks to lock it anyway, and a post-lock cleanup removes other plans for golfers that just became exhausted.
  • Duplicate-plan UX: planning a golfer already planned elsewhere now moves the plan instead of rejecting it.
  • Pre-lock visibility: leagues can opt into showPicksBeforeLock to reveal PLANNED picks to the league early.
  • planMultiple: one tap can plan a golfer across all of a user's leagues, skipping ineligible ones per league.

One reversal worth noting: getAllSeasonPicks now does apply read-side locking (the plan chose raw status); stale plans are instead surfaced by the timeline's own start-date check and largely removed at lock time by the cron cleanup.
