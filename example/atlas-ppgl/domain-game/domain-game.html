Domain

Picks & planner

What it isThe product core: one golfer per week, earn their real prize money, never reuse a golfer past the league's maxGolferUses. This domain owns the weekly pick, the season-long planner (prepicks), the auto-lock at tournament start, bot auto-picks, and the dashboard that fronts it all.
Pathsrc (routers/services/repositories pick*, dashboard*, bot-picks, planner + dashboard UI)
Files18
Entry pointstRPC pick.* and dashboard.get; two cron hooks called from golf-data's sync (lockPlannedPicksForTournament, autoPickForBots)
DataPick — one row per member × tournament, enforced by a DB unique constraint
Big idea
A pick is a soft reservation until its tournament tees off.

PLANNED picks are freely created, moved, and deleted all season — planning a golfer for a new week silently steals him from any other week he was planned for. Locking is dual: reads overlay LOCKED the instant startDate passes (applyReadSideLocking, no write needed), while the cron write-side pass (lockPlannedPicksForTournament) later promotes rows for real — validating field membership and exhaustion, and burning or returning stale plans per league policy.

Internal architecture

Two write paths into the pick table: the member path (pages → GolferPicker → pick router → service → repo) and the cron path from golf-data's sync (bot picks, then write-side locking inside the start-tournament transaction).

The dashboard repo is drawn folded into "pick repo" for brevity — it is a separate file (server/repositories/dashboard.ts) hitting the same tables.

☉The cron path deliberately avoids the server-only guard: pick-lock.ts and both bot-picks files import @/lib/db-core directly so golf-data's cron route and Vitest can call them outside a React server context.

In depth

pick service src/server

↑ back to cards

The router (routers/pick.ts) is 21 procedures of pure zod-validate-and-delegate; every rule lives in services/pick.ts. PLANNED mutations all share the same guardrails: league active, caller resolved to a leagueMemberId (via leagues), tournament exists and hasn't started, golfer exists. The signature move is in planPick/updatePlannedPick: if the chosen golfer is already PLANNED for a different week, that plan is deleted first — planning is a season-wide drag-and-drop, not per-week CRUD.

planPickMultiple fans one pick out across every eligible league the user is in (ACTIVE member, paid, current season), returning per-league planned/skipped results with reasons (exhausted, already_picked, planned_elsewhere). Unlike planPick it never auto-moves a conflicting plan, and it catches Postgres 23505 unique-violations to resolve concurrent-submit races as already_picked. It's feature-flagged in the UI by NEXT_PUBLIC_ENABLE_MULTI_LEAGUE_PICK.

Reads apply applyReadSideLocking — a pure overlay that reports any PLANNED pick whose tournament has started as LOCKED, so the UI is correct even before the cron write-side pass runs. findRemainingPool is the heaviest query: golfers not yet exhausted (LOCKED count ≥ league maxGolferUses, counting started-PLANNED picks), decorated with season earnings, missed cuts, per-golfer used/planned counts, and tri-state inField (true/false/null-when-field-unpublished).

Commissioner mutations (commissionerUpdatePick/RemovePick/AddPick) are the escape hatch: not-in-field and exhaustion are soft warnings rather than blocks (except add, which hard-blocks exhaustion), every change writes a CommissionerAction audit row in the same withTransaction, and if the tournament is already COMPLETED/FINALIZING the service calls scoring's recalculateSeasonForMember so standings update immediately.

pick locking src/server/repositories

↑ back to cards

lockPlannedPicksForTournament(tournamentId, season, executor) is the write-side half of the dual locking model. Golf-data's sync.ts calls it inside the same withTransaction that flips the tournament to IN_PROGRESS, so status change and lock promotion commit atomically.

For each PLANNED pick it checks two things: is the golfer a CONFIRMED entry in the tournament field, and is he not already exhausted (LOCKED count ≥ the league's maxGolferUses). Valid picks get status: LOCKED plus a lockedAt timestamp. A golfer not in the field forks on the league's burnStalePicks setting: burn mode locks the pick anyway (the week is spent on a golfer who never played), skip mode deletes it so the golfer returns to the pool. Exhausted golfers are skipped untouched.

After promotion there's a cleanup sweep: if locking made a golfer newly exhausted, his remaining PLANNED picks on other tournaments are deleted — the soft reservations collapse the moment the hard use is spent. Returns { locked, skipped, cleaned } for the cron log.

Fate of each PLANNED pick when its tournament starts. Read-side locking has already made these picks look LOCKED; this pass makes it true — or unwinds invalid plans.

bot picks src/server

↑ back to cards

autoPickForBots runs as step 1.5 of golf-data's runCronSync, deliberately before startTournaments so the picks it creates get locked in the same cron run. For every UPCOMING tournament that has a synced field, it finds bot users (user.isBot = true) whose league membership has no pick yet, filters the CONFIRMED field down to non-exhausted golfers for that member, and inserts a random PLANNED pick.

The insert uses onConflict (leagueMemberId, tournamentId) doNothing, leaning on the same unique constraint that protects human picks — a concurrent cron run can't double-pick a bot. Like pick-lock.ts, both files skip the server-only guard and import @/lib/db-core directly so the cron API route can use them.

dashboard src/server

↑ back to cards

One procedure, dashboard.get, returns everything the home screen needs. getDashboard decides a single isLive flag (tournament IN_PROGRESS, recently completed/finalizing, or past startDate) and then fetches in parallel: the member's pick for the current tournament, their standing (rank, total earnings, gap to the leader via scoring's findLeaderEarnings), either live league picks or the last completed result, and the next three UPCOMING tournaments overlaid with the member's pick status.

Two small read-model tricks: the current pick gets the same read-side PLANNED→LOCKED overlay as the pick service, and the league-picks list is windowed to 4 rows centered on the calling user (windowAround, exported for tests) so the dashboard shows your neighborhood of the standings, not the top of the table.

The server component (app/dashboard/page.tsx) renders the tournament bar, standing, last-result and upcoming cards, and mounts DashboardPickModule — a client component that shows the pick card (empty / planned / locked / not-in-field warning variants) and opens the shared GolferPicker drawer, refreshing server data on close.

planner & picker UI src/components/planner

↑ back to cards

/season is the prepick surface: PlannerTimeline renders the full schedule as upcoming/completed tabs, each row tappable to open GolferPicker in plan or updatePlan mode, with stale-plan detection (PLANNED but tournament started) and inline commissioner actions when viewing as commissioner. Server data arrives as initialData and stays live via TanStack Query.

GolferPicker is the one drawer for every pick mutation in the app — dashboard, league page, and planner all mount it. It searches/sorts the remaining pool (out-of-field golfers sink), and routes to pick.plan, pick.updatePlan, or the feature-flagged pick.planMultiple multi-league fan-out (toggle persisted in localStorage, per-league result list rendered after submit).

RemainingPool + UsageCircles visualize the season budget: every unexhausted golfer with events/MC/earnings stats and one circle per allowed use — filled (used), half (planned), empty (available). /season/[userId] is the read-only member view: LOCKED-only history for peers (commissioners see true DB status plus MISSED weeks so they can backfill), with the pool's planned counts hidden unless the league opts into showPicksBeforeLock.

Data it owns

One Prisma model — but it is the table the whole game turns on.

PickOne row per league member × tournament. @@unique([leagueMemberId, tournamentId]) is the one pick per week rule enforced at the database — both planPickMultiple's race handling and the bot inserter's onConflict doNothing lean on it.
Pick.status (PickStatus)PLANNED | LOCKED. There is deliberately no MISSED or BURNED status — missed weeks are the absence of a row (computed by findStartedTournamentsWithoutPicks), and burned picks are just LOCKED picks whose golfer never played.
Pick.lockedAtStamped only by the write-side lock (pick-lock.ts) or a commissioner backfill on a started tournament — read-side locking never writes it.

Everything else it touches is borrowed: Tournament, Golfer, TournamentField, TournamentResult belong to golf-data; League, LeagueMember, CommissionerAction to leagues; LeagueMemberSeason rollups to scoring.
