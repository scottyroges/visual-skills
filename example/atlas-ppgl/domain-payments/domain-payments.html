Big idea
There is no payments table — the League row is the ledger.

The league's identity rides inside PayPal's custom_id order metadata, so capture and webhook can both recover which league was paid for statelessly. All payment state is paymentStatus (FREE → PENDING → PAID) on the League row plus a use counter on PromoCode, flipped by one idempotent confirmPayment reachable from two redundant paths.

Checkout & webhook flow

The commissioner pays through PayPal Buttons rendered by the leagues create-form. Confirmation is deliberately redundant: the client calls league.captureOrder itself, and PayPal independently posts a PAYMENT.CAPTURE.COMPLETED webhook. Both funnel into the same idempotent confirmPayment in the leagues service.

Whichever path lands first wins: confirmPayment no-ops if the league is already PAID, so the client capture and the webhook can race safely. A user who closes the tab after approving still gets their league unlocked by the webhook.

In depth

Payment orchestration src/lib/payment.ts

↑ back to cards

One 35-line file that keeps PayPal out of the rest of the codebase. It defines the PaymentProvider interface — createCheckout({ leagueId, userId, price? }), captureOrder(orderId), verifyWebhook({ body, headers }) — plus the result shapes each returns. WebhookResult is a discriminated union (invalid_signature | ignored | completed), which is what lets the webhook route map outcomes to HTTP statuses without knowing anything about PayPal events.

It also owns the list price as code, not data: LEAGUE_CREATION_PRICE (a string, because that is PayPal's amount format) and LEAGUE_CREATION_CURRENCY. The leagues service derives everything else from it — promo discount math is Math.max(0, FULL_PRICE - discount) over parseFloat(LEAGUE_CREATION_PRICE). createCheckout accepts an optional price override precisely so partially-discounted checkouts can charge less than list.

PayPal integration src/lib/paypal.ts

↑ back to cards

paypalProvider implements the contract against PayPal REST (v2 Checkout Orders + v1 webhook verification), switching between the sandbox and live base URL on NODE_ENV. It authenticates with client-credentials OAuth (PAYPAL_CLIENT_ID / PAYPAL_CLIENT_SECRET env vars) and caches the access token at module level, expiring it 60 seconds early; _clearTokenCache() exists only so tests can reset that state. The file is server-only — it can never leak into a client bundle.

The load-bearing trick is metadata: createCheckout JSON-encodes { leagueId, userId } into the purchase unit's custom_id (kept under PayPal's 127-char limit because both are UUIDs). captureOrder and verifyWebhook decode it back out, which is why the app needs no orders table — PayPal carries the correlation. verifyWebhook posts the transmission headers back to PayPal's verify-webhook-signature endpoint (using the PAYPAL_WEBHOOK_ID env var), rejects anything unverified, ignores every event type except PAYMENT.CAPTURE.COMPLETED, and returns the decoded leagueId plus the order id as sessionId.

Webhook route src/app/api/paypal/webhook/route.ts

↑ back to cards

A 19-line Next.js route handler, and deliberately so. It reads the request body as raw text (the signature is verified over the exact bytes, so no JSON middleware may touch it first), flattens the headers into a plain record, and hands both to handleWebhookPayment in the leagues service. invalid_signature maps to a 401; everything else — including events the app ignores — returns 200 { received: true } so PayPal stops retrying.

This route is the safety net, not the primary path. Normally the create-league form calls league.captureOrder the moment the buyer approves, and that capture confirms payment synchronously (after checking the caller is the commissioner and that the captured order's embedded leagueId actually matches the league being paid for). The webhook covers the gap where the user approves in the PayPal popup and then closes the tab: PayPal's async event still reaches confirmPayment, which flips paymentStatus to PAID and redeems any attached promo — and no-ops if the client capture already won the race.

Promo codes src/server/repositories/promo-code.ts

↑ back to cards

The repository is a single function: findByCode uppercases the input and returns the code row (uses, discount, expiry, active flag). The leagues service does the judging — validatePromoCode (behind league.validatePromoCode, called live from the create-league form) checks active / usage-limit / expiry and returns the computed discount, and createWithPromo re-checks at creation time.

A code's discountAmount drives two different flows. Null means free: the league is inserted already PAID, and the redemption counter is incremented in the same transaction via a guarded UPDATE (it re-checks active, max-uses, and expiry inside the WHERE clause, so two concurrent redemptions of a nearly-exhausted code can't both succeed). A dollar value means partial discount: the league is inserted PENDING with promoCodeId attached, a PayPal checkout is created at the reduced price, and redemption is deferred until confirmPayment — so an abandoned discounted checkout never burns a use.

Admin CRUD lives at /admin/promos: a list page, a create page, and a detail page that doubles as the edit form plus a redemption history (leagues created with that code, with commissioner and date). The pages are thin — they call admin.* tRPC procedures whose service and repository belong to the admin domain; the shared PromoForm client component drives both create and update mutations, constraining partial discounts to whole dollars below the list price.

Data it owns

One model of its own, plus a three-column footprint on a model owned by leagues.

PromoCodeUnique code, optional maxUses vs currentUses counter, optional discountAmount (null = free code, value = whole-dollar discount), optional expiresAt, active flag.
League.paymentStatusThe actual payment ledger — FREE | PENDING | PAID enum on the leagues-owned League model. PENDING leagues are blocked by assertLeagueActive until paid.
League.paymentSessionIdThe PayPal order id recorded at confirmation — the only persisted trace of the transaction.
League.promoCodeIdOptional FK to PromoCode; also how the admin detail page reconstructs a code's redemption history (leagues that used it).

There is deliberately no Order/Payment table. Correlation between a PayPal order and a league travels inside the order's custom_id metadata, and the terminal state lands on the League row.

Seams

The domain's public surface is one HTTP route, a handful of tRPC procedures served by neighboring routers, and the paypalProvider object the leagues service consumes in-process.
