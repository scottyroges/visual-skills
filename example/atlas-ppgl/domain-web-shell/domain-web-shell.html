publicProcedure → protectedProcedure (throws UNAUTHORIZED without a session) → adminProcedure (throws FORBIDDEN unless role === "admin") — every API call climbs exactly as far as it must.

Internal architecture

The request path. Two entry doors — client components speak HTTP to /api/trpc, server components call the router in-process via serverTRPC — and both converge on createTRPCContext, the procedure ladder, and the same router → service → repository → Postgres stack.

Both doors share one context: createTRPCContext resolves the Better Auth session from request headers, then the ladder decides how far the call may climb. Server pages never pay an HTTP round-trip — serverTRPC uses createCallerFactory(appRouter).

☉Everything below the ladder belongs to the neighbor domains — web-shell owns the pipe, not the water.

In depth

tRPC plumbing src/server · src/lib/trpc · src/app/api/trpc

↑ back to cards

server/trpc.ts is the smallest and most load-bearing file in the repo. createTRPCContext calls auth.api.getSession (Better Auth, from the accounts domain) against the incoming headers and yields { session, userId }. initTRPC is configured once with the superjson transformer — matching the transformer on the client link, so Date/Decimal values survive the wire.

The ladder is two middleware steps: protectedProcedure throws UNAUTHORIZED when ctx.userId is null (and narrows its type to string for everything downstream), and adminProcedure extends it, throwing FORBIDDEN unless the session user's role is "admin". Every one of the 11 domain routers builds its procedures from one of these three rungs.

routers/_app.ts is pure aggregation: it merges auth, health, tournament, golfer, pick, dashboard, league, leaderboard, admin, earningsAdjustment, and user into appRouter and exports the AppRouter type — the single type that powers end-to-end inference in every client component. Notably there is no payments router: PayPal talks over plain API routes, so payments is absent from this file.

Consumption is two-sided. lib/trpc/client.tsx (client) builds a TanStack Query QueryClient (30s staleTime, browser-singleton) and a tRPC client with a single httpBatchStreamLink to /api/trpc; createTRPCContext<AppRouter>() from @trpc/tanstack-react-query yields the useTRPC hook. lib/trpc/server.ts (server) skips HTTP entirely: createCallerFactory(appRouter) plus a fresh context gives serverTRPC(), the in-process caller used by 27 server pages and the NavBar.

Each rung is a middleware use() on the previous one — auth checks live here once, never in routers.

db core src/lib

↑ back to cards

lib/db-core.ts builds the singleton Kysely<DB> over a pg.Pool on DATABASE_URL, with CamelCasePlugin (snake_case columns → camelCase properties) and the generated types from @/generated/kysely/types. In non-production it caches the instance on globalThis so Next.js hot reloads don't leak pools. The pool registers a swallow-all error handler — when Vercel freezes a serverless function, idle connections get killed, and without the handler that surfaces as an uncaught exception.

withTransaction(fn) is the entire transaction API: it runs fn inside db.transaction().execute() and hands the transaction executor to the callback. The convention (enforced by review, visible in every service) is that repository functions accept an optional executor: Kysely<DB> = db parameter, so a service can pass the transaction through multiple repository calls atomically.

lib/db.ts is a two-line gate: import "server-only" then re-export { db, withTransaction } from db-core. App code imports @/lib/db and gets a build error if the import ever leaks into a client bundle; the 14 seed scripts under prisma/seed/ import @/lib/db-core directly because they run outside Next.js where server-only would throw.

app chrome src/app · src/components

↑ back to cards

app/layout.tsx (RootLayout) loads two Google fonts as CSS variables (Literata + Hanken Grotesk), declares the site-wide metadata (title template, Open Graph, Twitter card), imports the flag-icons stylesheet, and wraps every page in Providers → SessionRefresh → NavBar → children. Providers is a one-liner that mounts TRPCReactProvider; SessionRefresh comes from accounts.

NavBar is an async server component: it calls serverTRPC() on every render, checks auth.me(), and returns null when signed out — which is why the landing page renders its own header with a Sign In link. Signed in, it fetches league.myLeagues(), resolves the active league via getActiveLeagueId (from leagues), and renders the LeagueSwitcher, desktop links (Dashboard / Season / Standings / Leagues), and the same links again inside MobileMenuToggle.

MobileMenuToggle is the only client component here: a hamburger button + slide-out panel with two close behaviors (any link/button click inside the menu, any click outside it), tracked with refs and aria-expanded.

app/page.tsx (Home) is the signed-out landing page — it redirects authenticated users to /dashboard, then fetches tournament.current() and the top-10 of leaderboard.global() (both .catch-guarded so an empty DB still renders) and lays out hero, how-it-works, live leaderboard teaser, league pitch, and footer.

shared utils src/lib

↑ back to cards

lib/constants.ts is a single well-known ID: GLOBAL_LEAGUE_ID = "ppgl-global", the league every user is auto-enrolled in. The scoring leaderboard service and leagues league service branch on it, and UI in leagues/admin special-cases it.

lib/format.ts is where display rules encode game rules. formatCurrency renders whole-dollar USD; formatPosition does ordinal suffixes (with the 11th/12th/13th trap handled); formatScore renders golf score-to-par (E, +3, -5, – for null). formatPurse prefers the actual purse and falls back to ~$X for estimates. formatDate pins America/New_York so tournament dates don't shift per viewer timezone.

isMissedCut is the most game-aware helper: a pick is a missed cut only when it earned exactly $0 with no real finish position and is LOCKED — and never while the tournament is IN_PROGRESS, because in round 1 everyone has $0 and a null position; the cut simply hasn't happened yet.

Data it owns

None — and that's the point. web-shell is pure plumbing: it owns the connection to the database, not any table in it.

(none)No Prisma models attribute here. The Kysely db singleton and withTransaction are owned here, but every table they touch belongs to a neighbor domain.

If you're looking for a model, start from the feature domain that owns it — e.g. picks in game, tournaments in golf-data.

Seams

web-shell's surface is the contract every other domain builds on: the procedure rungs, the two tRPC doors, the transaction primitive, and the shared formatters.

↑ Exposes
  • publicProcedure — Unauthenticated rung — landing page reads, health.
  • protectedProcedure — Session-required rung; the workhorse for all member features.
  • adminProcedure — Role-gated rung for the back-office.
  • appRouter — The composed API; its `AppRouter` type drives end-to-end inference.
  • POST /api/trpc — The single HTTP endpoint (also `GET`), batched + streamed + superjson.
  • serverTRPC — In-process caller — server components skip HTTP entirely.
  • useTRPC — Client hook (TanStack Query integration) from `lib/trpc/client.tsx`.
  • withTransaction — Services' only way to make multi-repository writes atomic.
  • db — Kysely singleton — repositories only (plus `auth.ts` and `health.ts` by design).
  • formatCurrency — Representative of the `format.ts` family used across every UI domain.
  • GLOBAL_LEAGUE_ID — Well-known global-league ID consumed by scoring/leagues services.
