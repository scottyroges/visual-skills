Start here

A season-long fantasy golf game — pick one PGA Tour golfer a week, earn his real prize money, never reuse a golfer — built as a Next.js + tRPC monolith whose heartbeat is a 15-minute ESPN sync cron.

WhatPPGL (Perfect Pick Golf League, ppgl.app): private leagues with PayPal checkout, a season planner for future-week prepicks, live projected earnings during tournaments, and commissioner + admin override surfaces. One Vercel deployment, one Postgres (Neon).
ShapeA strict three-layer server — routers (thin, validate → service) → services (business logic, withTransaction) → repositories (the only layer that touches db) — under a Next.js App Router front. Eight domains: five product slices, a data layer, an ops surface, and the shared shell.
SpineThe weekly loop: a user plans a PLANNED pick → the cron locks it when the tournament starts (field + not-yet-used validation) → live sync writes projected earnings all weekend → completion promotes real money and triggers the earnings rollup → standings re-rank.
1
The cron is the heartbeat.
GET /api/cron/sync runs every 15 minutes and drives every game-state transition: it locks PLANNED picks at tee-off, projects live earnings, promotes final results, kicks the earnings rollup, runs bot auto-picks, sends pick reminders, and cleans up abandoned leagues. Nothing else moves the game clock.
2
Picks are soft until they're hard.
A PLANNED pick is a free-floating intention — movable, deletable, not consuming the golfer. Only the LOCKED status burns a golfer for the season. Reads overlay the lock the moment startDate passes; the cron makes it real in the database shortly after.
3
Membership is the permission system.
resolveLeagueMemberId + assertLeagueActive gate every gameplay mutation, and the leagueMemberId they return is the key picks, standings, and earnings all hang off. Everyone is lazily auto-enrolled in the global league on first contact, so the guard doubles as onboarding.
4
Money is recomputed, never incremented.
LeagueMemberSeason is a materialized rollup: every trigger recomputes a member's whole season from one SQL aggregate over LOCKED picks (plus commissioner adjustments) and overwrites the row. There is no ledger to drift.
5
Auth checks live in exactly one place.
The three-rung procedure ladder in src/server/trpc.ts — publicProcedure → protectedProcedure → adminProcedure — is the only authorization boundary; page-level redirects are UX, not security.

The domain map

8 domains

Five product slices (game, leagues, scoring, accounts, payments) over a data layer (golf-data) and the shared shell (web-shell), with admin as an operator lens that writes into all of them. Arrows are the load-bearing runtime edges — every domain additionally rides web-shell's tRPC/db plumbing (those edges are omitted for legibility), and admin reaches into every slice.

The heartbeat — its cron drives every game-state transitionExternal world (ESPN, PayPal)Shared plumbing every domain rides

golf-data is highlighted because its 15-minute cron is the game clock: pick locking, live money, finals, rollups, reminders, and league cleanup all fire from it. payments is the one domain not mounted on the tRPC root router — PayPal runs over plain API routes.

The weekly loop

runtime

The one flow worth memorizing — what happens to a single pick across a tournament week. Everything left of the lock is user intent; everything right of it is machine-driven by the sync cron.

A stale PLANNED pick (golfer not in field, or already used) is not burned — it's left behind and the member simply earns $0 that week, per the league's burnStalePicks policy. Missed weeks are the absence of a row, not a status.
