Big idea
Membership is the permission system — every other gameplay domain answers "may this user act here?" by calling into leagues.

resolveLeagueMemberId and assertLeagueActive (src/server/services/league.ts) are imported by the pick and dashboard services; the leagueMemberId they return is the identity key that picks, standings, and earnings all hang off. Everyone is lazily auto-enrolled in the global league (ppgl-global), so the guard doubles as onboarding.

Internal architecture

One thin tRPC router in front of one fat service. The service is also entered from the side — by the PayPal webhook, by the cron sync's cleanup, and by other domains' services importing its guards directly.

league service is the hub: fronted by the router, but also called directly by the payments webhook, the golf-data cron, and the game domain's services. The commissioner UI bypasses it entirely — those mutations belong to the pick router.

☉Note the two side doors into the service: handleWebhookPayment (from src/app/api/paypal/webhook/route.ts) and cleanupAbandonedLeagues (from src/server/services/cron-sync.ts). Neither goes through tRPC.

In depth

league service src/server/{routers,services,repositories}/league.ts

↑ back to cards

routers/league.ts is deliberately thin — 16 procedures that only zod-validate and delegate (league name 3–50 chars, maxGolferUses 1–3, everything else booleans and ids). All 16 are protectedProcedure except league.preview, which is public so an invite link can render a league preview to signed-out visitors.

services/league.ts holds the real rules. Creation inserts League + commissioner LeagueMember + LeagueMemberSeason atomically via withTransaction (shared helper insertLeagueWithMember). Joins are race-safe twice over: the 20-member cap (MAX_LEAGUE_SIZE) is re-counted inside the transaction to kill the TOCTOU window, and a concurrent double-join is caught by the @@unique([userId, leagueId]) constraint then re-queried. The same catch-requery pattern makes ensureGlobalLeagueMembership idempotent — global-league membership is created lazily the first time any league-scoped call runs.

Member lifecycle: joining an open league → ACTIVE; with requireApproval on → PENDING (no season row until approved, so pending members never appear in standings). approveMember re-checks the cap, flips to ACTIVE, inserts the season row, and logs an APPROVE_MEMBER audit action. rejectMember hard-deletes the row — the code comments that no audit entry is possible because CommissionerAction.targetMemberId has an FK to the member being deleted. removeMember soft-removes (REMOVED) but scrubs the member's planned picks, season rows, and earnings adjustments; locked picks survive. Removed members can rejoin (back through approval if required).

Payment gating: privately created leagues are born paymentStatus: PENDING and inert — assertLeagueActive rejects picks and joinLeague rejects joins until PayPal capture (client captureOrder or webhook handleWebhookPayment) flips them to PAID via confirmPayment, which also redeems any deferred partial-discount promo. cleanupAbandonedLeagues deletes PENDING leagues older than 48h; the repo re-checks paymentStatus inside its delete transaction to avoid racing a late webhook.

LeagueMemberStatus transitions in services/league.ts. Only ACTIVE members get a LeagueMemberSeason row — which is why pending members are invisible to standings.

create & checkout src/app/league/{create,created}

↑ back to cards

CreateLeagueForm is a single client component handling three payment paths. No promo: league.create makes a PENDING league, then league.createCheckoutSession fetches a PayPal order id and the form swaps to an embedded PayPalButtons checkout (@paypal/react-paypal-js); on approval it calls league.captureOrder and routes to /league/created. Free promo: league.createWithPromo redeems the code inside the creation transaction and the league is born PAID — straight to the success page. Partial-discount promo: league is PENDING with promoCodeId set, checkout runs at the discounted price, and redemption is deferred to payment confirmation.

The form also collects the rule settings that the rest of the game reads back later: format (maxGolferUses — "One and Done" vs "Double Use"), burnStalePicks, showPicksBeforeLock, and requireApproval. /league/created reads ?id=&code= and shows the copyable invite URL (/league/join?code=…) — the commissioner's first job is distribution.

Three routes to a live league. A PENDING league is fully inert — no joins, no picks — until capture; abandon it and the cron deletes it after 48 hours.

join & approval src/app/league/join + app/league/[id]

↑ back to cards

/league/join?code=… is the invite landing page and it works signed-out: the server page calls public league.preview for name + member count, then branches on auth and membership state. Signed-out visitors get sign-up/sign-in links that carry ?invite=<code> through the auth flow; existing members get a "Go to League" shortcut; unpaid leagues show an "awaiting payment" notice instead of a join button.

JoinLeagueClient fires league.join and renders the outcome: ACTIVE → "You're In!" with a How-to-Play CTA; PENDING → "Request Sent" interstitial. On the approval side, PendingMembers (rendered on the league page only for the commissioner of a requireApproval league) polls league.pendingMembers and mutates league.approveMember / league.rejectMember, then invalidates the query and router.refresh()es so the server-rendered header badge and standings update together. Meanwhile a pending member visiting /league/[id] sees a blocked interstitial that points them at the global league while they wait.

league pages src/app/leagues + src/app/league/[id]

↑ back to cards

/league/[id]/page.tsx is the biggest page in the app — a server component that fans out in Promise.all to five other-domain procedures: leaderboard.byLeague (season standings), pick.leagueCurrentWeek (this week's picks), league.auditLog, tournament.schedule, and leaderboard.weeklyStandings when a ?week= param selects a past tournament. A local leagueHref helper threads page/twPage/week params through every pagination and navigation link, and WeekSwipeWrapper adds mobile swipe-to-navigate between weeks on top of the arrow links.

The page renders differently by viewer: pending members get an interstitial, commissioners get the pending-members queue and an unpaid-league banner with a "Complete Payment" link back into checkout, private-league members get the invite-link card and the commissioner-actions audit log. The inline pick table swaps between a "planned" view (before lock, only if showPicksBeforeLock or picks are the viewer's own — enforced server-side in game's service) and a live/final earnings view.

/league/[id]/rules is a settings-aware how-to-play: its withdrawal, visibility, and joining sections change copy based on the league's burnStalePicks / showPicksBeforeLock / requireApproval flags. /league/[id]/settings lets the commissioner edit those same flags via league.updateSettings (non-commissioners see a read-only form) and manage the roster — LeagueMembers confirms then fires league.removeMember, warning that planned picks will be deleted.

league context src/lib/league-context.ts + src/components/league

↑ back to cards

The active league is a single httpOnly cookie (ppgl-active-league, 1-year max-age) managed by two server actions in lib/league-context.ts. getActiveLeagueId validates the cookie against the user's actual leagues (falling back to the first league) and honors a searchParams override for deep links — it deliberately never writes during render, because Next.js only allows cookie writes in server actions and route handlers.

Two writers exist: LeagueSwitcher, the nav-bar <select> (rendered by web-shell's nav-bar.tsx from league.myLeagues), which sets the cookie in a transition then either pushes to the new league's page or refreshes in place — it also smuggles a "+ Create League" option into the dropdown; and SyncLeagueContext, a render-nothing client component mounted on /league/[id] that syncs the cookie once on mount, so simply visiting a league makes it your active one. Dashboard, season, tournament, and member pages all read the same cookie to scope their queries.

commissioner tools src/components/commissioner

↑ back to cards

These widgets let a commissioner fix someone else's pick — the escape hatch for "my buddy forgot to pick before lock." CommissionerToggle is pure DOM: it flips a data-commissioner-active attribute on the nearest [data-commissioner-container], and CSS reveals the per-row action buttons. No server state; commissioner-ness was already decided server-side by whichever page rendered the toggle.

CommissionerPickActions (edit/remove an existing pick) and CommissionerAddPick (add a missing one) share a pattern: local golfer typeahead over a pre-fetched list, a live pick.commissionerValidate query surfacing warnings (already used, exhausted uses — exhaustion hard-blocks the Add button), then pick.commissionerUpdate / pick.commissionerRemove / pick.commissionerAdd. Those mutations live in the game domain's pick service, which enforces commissioner authorization again server-side and writes the CommissionerAction rows this domain's audit log displays. The components are mounted from game-domain surfaces: the season page (app/season/[userId]/page.tsx) and the planner timeline.

Data it owns

The membership graph and its paper trail. All writes go through repositories/league.ts.

LeagueName, unique inviteCode, season, type (GLOBAL/PRIVATE), commissionerId, the rule flags (maxGolferUses, burnStalePicks, showPicksBeforeLock, requireApproval), and payment state (paymentStatus, paymentSessionId, promoCodeId).
LeagueMemberuser ↔ league with status (ACTIVE/PENDING/REMOVED) and @@unique([userId, leagueId]) — the constraint the join/global-enroll race handling leans on. Its id (leagueMemberId) is the key picks and earnings hang off.
LeagueMemberSeasonPer-member-per-season row created here on join/approval; the totalEarnings/totalPicks/totalMissedCuts rollups on it are written by scoring's earnings pipeline.
CommissionerActionAppend-only audit log: action string + details JSON (golfer/tournament ids resolved to names at read time). Written here (approve/remove) and by game's pick service + admin.

PromoCode is referenced (League.promoCodeId) and redeemed here (redeemPromoCode guards uses/expiry in SQL), but the model and its CRUD belong to payments. Pick belongs to game — this domain only deletes PLANNED picks when removing a member.

One tRPC router out; two service functions that act as the app-wide membership guard; five domains leaned on.
