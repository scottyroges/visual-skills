The one-way clock
ESPN is the single source of truth and data flows one direction only: a 15-minute Vercel cron pulls schedule, field, and leaderboard data, upserts it keyed on espnId, and drives every game-state transition — locking picks when a tournament starts, projecting earnings from the payout table while it's live, and promoting real prize money into standings only when ESPN's core API finally reports it.

The FINALIZING status is the trick: ESPN flips an event to STATUS_FINAL hours before it posts real earnings, so the cron parks the tournament there (keeping projections in standings) and retries every run until the winner's earnings are non-zero.

Internal architecture

The hero flow: every 15 minutes Vercel hits the cron route, which runs the 6-step pipeline. The sync service is the hinge — it pulls from ESPN, writes through the upsert repos, and pushes side effects into the game and scoring domains.

Reads-side (tournament.* / golfer.* tRPC) is deliberately off this diagram — it only ever queries Postgres; nothing user-facing calls ESPN.

☉Step order inside runFullSync is load-bearing: completion checks run before live standings so a finished tournament's final results are never overwritten with projections, and bot auto-picks run before startTournaments so they get locked with everyone else's.

In depth

ESPN provider src/server/services/golf-data

↑ back to cards

types.ts defines the provider-agnostic contract (GolfDataProvider with getSchedule / getField / getResults / getEarnings / getEventStatus / getEventDetails) plus the neutral data shapes (TournamentData, GolferFieldEntry, TournamentResultData, EarningsData). The rest of the app only ever sees these — a future Data Golf provider would slot in behind the same interface.

espn/client.ts implements it as a singleton EspnProvider class over two ESPN endpoints: the site scoreboard (site.api.espn.com, has athlete names/scores/linescores but no money) and the core API (sports.core.api.espn.com, has purse + actual earnings per competitor, but everything else is $ref links). getEarnings merges the two by competitor id. All fetches carry a 10-second AbortSignal timeout, and the season schedule is cached in-process per season year.

A quirk worth knowing: ESPN has no per-event competitor endpoint the client uses — fetchCompetitors looks up the event's date from the cached schedule, refetches the scoreboard for that date (?dates=YYYYMMDD), then finds the event in the response. TBD/placeholder entries without athlete.displayName are filtered out.

espn/mappers.ts is where ESPN's mess is normalized. Ties: ESPN's order field is sequential even for tied players (and can reflect tee times during live play), so results are re-grouped by score string and assigned true tied positions with tiedCount. Made cut: players with fewer than 3 linescores missed the cut — unless nobody has reached round 3 yet, in which case the cut hasn't happened and everyone is in play. Dates: ESPN ISO timestamps carry arbitrary UTC times that shift the calendar day in US timezones, so parseEspnDate pins every date to 17:00 UTC (~noon ET).

Neither ESPN API is sufficient alone: the scoreboard has who/where, the core API has the money. getEarnings joins them by competitor id.

Payout table src/server/services/golf-data

↑ back to cards

A hardcoded table of standard PGA Tour payout percentages for 36-hole-cut events — positions 1 through 65, winner 18.000%, last paid spot 0.215% — with calculateEarnings(purse, results) on top. Tie handling matches real tour rules: N players tied at position P split the average of the payouts for positions P through P+N−1. Missed-cut players get $0.

This is projections only. It runs during syncLiveStandings to give in-progress tournaments believable dollar figures (the UI labels them "est."); the moment a tournament completes, syncResults overwrites everything with ESPN's actual earnings. Known accepted gaps: majors and Signature Events pay 20% to the winner, no-cut events redistribute differently, and amateurs (who earn nothing) aren't flagged in scoreboard data.

server/services/golf-data/payout.tsPAYOUT_PERCENTAGES (1-indexed, 65 paid positions) + calculateEarnings with tie averaging.

calculateEarnings(purse, results) → Map<espnId, dollars> — tie-averaged, missed cut ⇒ 0.

Sync pipeline src/server/services

↑ back to cards

services/sync.ts holds the five operations the cron composes, each idempotent by construction because every write goes through espnId-keyed upserts: syncFields (upsert golfers + field entries, backfill the purse), syncLiveStandings (upsert TournamentResult rows with projected earnings from the payout table), syncResults (upsert real earnings, flip to COMPLETED, trigger rollup), checkForCompletedTournaments (poll ESPN status for every IN_PROGRESS event), and startTournaments (flip UPCOMING → IN_PROGRESS).

The status flip in startTournaments is the one place this domain uses withTransaction: the status update and lockPlannedPicksForTournament (game domain) commit atomically, so a failed lock can never leave picks editable on a tournament that's already live.

syncResults implements the FINALIZING dance: if ESPN reports the event final but the winner's earnings are still $0, the tournament parks in FINALIZING — projections stay in place, rollup still runs so standings show projected money, and every subsequent cron run retries until real earnings appear.

repositories/sync.ts and repositories/upserts.ts are the write-side repos. Notably they import db-core (not lib/db) and skip the server-only guard, because the cron route and tsx-run seed scripts execute outside React server component context. upserts.ts is also reused verbatim by the prisma/seed/* backfill scripts. updateTournamentStatus sets completedAt via coalesce(completed_at, now) so only the first completion transition stamps the time.

The TournamentStatus lifecycle as the cron drives it (CANCELLED and the admin excluded flag sit outside the automated path).

Cron entry src/app/api/cron/sync

↑ back to cards

runFullSync in services/cron-sync.ts is the orchestrator: (1) complete finished tournaments (before live sync, so finals never get clobbered by projections), (1b) retry FINALIZING events, (1.5) auto-pick for bot members (before starts, so bot picks get locked), (2) start due tournaments, (3) live standings for everything in progress, (4) field sync for events starting within 7 days, (5) clean up abandoned PENDING leagues, (6) send pick-reminder emails for events starting within 16 hours.

Every step is individually try/caught into a results bag — one failing step never stops the rest. Any key ending in Error downgrades the run to status: "partial", and the whole result JSON is written to cron_log fire-and-forget via logCronRun.

The HTTP wrapper at app/api/cron/sync/route.ts checks Authorization: Bearer <CRON_SECRET>, sets maxDuration = 300 for Vercel, and returns 200 for a clean run or 207 Multi-Status for a partial one. vercel.json schedules it every 15 minutes (*/15 * * * *).

Tournaments src/app/tournament

↑ back to cards

The read-side tournament vertical follows the house layering exactly: routers/tournament.ts (thin, 4 procedures) → services/tournament.ts → repositories/tournament.ts. getCurrent picks what the dashboard shows with a deliberate priority: in-progress first (by status, since ESPN end dates can be midnight ET and date math misses Sunday afternoons), then anything completed within the last 24 hours (so users see final results before being rushed to next week), then the next upcoming event.

getDetail powers /tournament/[id]: field + results join, plus a league pick overlay — but only after validating the caller is actually a member of the requested league, and never for GLOBAL leagues (too many members to be useful). Once a tournament has started, the field re-sorts from world ranking to finish position → earnings → score.

The page itself branches on status: upcoming shows the field with the game domain's pick module embedded; live/completed shows the leaderboard with "est." earnings labels and the scoring domain's weekly standings table. Missed cut is derived in the UI ($0 earnings + no finish position, only after someone has a real position) — it is not stored.

Golfers src/app/golfer

↑ back to cards

Same vertical shape, smaller: routers/golfer.ts exposes byId (profile + stats) and listAll (id + name ordered by world ranking, feeding pickers). services/golfer.ts composes the profile from two parallel repo calls scoped to getCurrentSeason().

getSeasonStats is one aggregate SQL query — events played, total earnings, best finish, and a missed-cuts heuristic: $0 earnings + no/zero finish position on a COMPLETED tournament counts as a missed cut (which also absorbs WD/DQ; the schema can't distinguish them). In-progress tournaments are excluded so everyone doesn't look cut mid-Thursday.

/golfer/[id] is a public page: avatar, flag, world rank, season stat rows, and a season results table with live rows badged and estimated earnings labeled.

Display helpers src/components

↑ back to cards

The small shared vocabulary every golf-facing surface uses. GolferAvatar renders an ESPN CDN headshot (a.espncdn.com/i/headshots/golf/players/full/<espnId>.png via golferHeadshotUrl) in five sizes, falling back on error or missing espnId to initials on a deterministic name-hashed oklch background.

CountryFlag maps ESPN's full English country names ("United States", "South Korea", and sub-national "England"/"Scotland" → gb-eng/gb-sct) to flag-icons CSS classes via the CODE_MAP lookup in lib/country-flag.ts; unknown countries render nothing rather than a broken flag.

lib/season.ts is deliberately trivial — getCurrentSeason() returns the calendar year — but it is the single source of truth for "current season" across golfer stats, schedules, and the scoring domain's rollups. When PPGL outlives the Jan–Aug PGA season assumption, this is the one place to fix.

Data it owns

The golf-world half of the schema — everything keyed to ESPN identity. Game-side models (Pick, LeagueMemberSeason) reference these but belong to their own domains.

TournamentOne schedule row per event, espnId unique. Carries the TournamentStatus lifecycle (UPCOMING → IN_PROGRESS → FINALIZING → COMPLETED, plus CANCELLED), purse (confirmed from ESPN) vs estimatedPurse (manual fallback), completedAt (stamped once), and an admin excluded flag every read query filters on.
GolferespnId unique, name, country (ESPN's full English name), nullable worldRanking. Continuously refreshed by field/result upserts.
TournamentFieldWho's playing: unique (tournamentId, golferId), FieldStatus CONFIRMED/WITHDRAWN. Synced daily-ish for events starting within 7 days.
TournamentResultUnique (tournamentId, golferId): finishPosition (tied players share it), earnings, scoreToPar, rounds. The same row holds projected earnings while live and actual earnings after completion — consumers can't tell from the row alone; tournament status says which.
CronLogOne row per runFullSync execution: full JSON response blob, success flag, durationMs. Read by admin/health surfaces.

Missed cut is intentionally not stored — both the golfer stats query and the tournament page derive it ($0 earnings + no finish position on a completed event), which also silently absorbs WD/DQ.

Seams

Six public tRPC procedures plus the secret-guarded cron route in; five neighbor domains leaned on out. This domain also has heavy inbound traffic that isn't a dependency: game, scoring, and admin import its repos and services directly.
