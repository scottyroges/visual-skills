Big idea
Identity is rented, not built — Better Auth owns every credential path against the app's own Postgres.

Sign-up, sign-in, password change/reset, and username updates all flow through authClient → the /api/auth/[...all] catch-all → the betterAuth instance in server/auth.ts. The rest of PPGL only ever asks who is this? — createTRPCContext calls auth.api.getSession per request. The only custom identity logic the app carries is a profanity-gated username validator and its own email templates.

Internal architecture

Two independent halves share one email gateway. The interactive half: browser forms talk to authClient, which hits the Better Auth catch-all route; the betterAuth instance persists users/sessions in Postgres and every tRPC request re-derives the session from it. The background half: golf-data's cron fans into the notification service, which queries eligibility and batch-sends through Resend.

The betterAuth instance is the hub: browser mutations reach it over HTTP, while every tRPC call reads it server-side through auth.api.getSession.

☉Note what is missing: there is no accounts service layer for credentials. Better Auth is the service — the app's own router/service/repo stack in this domain exists only for the notification preference, and the notification pipeline deliberately skips server-only guards so cron routes can call it.

In depth

Better Auth core src/server · src/lib · src/app/api/auth

↑ back to cards

server/auth.ts builds the single betterAuth instance: email + password auth against the app's own Postgres (database: { db, type: "postgres" } — the same Kysely instance the repos use, so users and sessions live next to picks and leagues, no separate identity store). Cookie sessions are long-lived and periodically re-stamped (session.expiresIn / updateAge), rate limiting is on by default (an env flag disables it for tests), and trustedOrigins is derived from the base URL with the www./apex twin added so both hosts pass CSRF origin checks.

Two plugins do the customization: username() enforces 3–20 chars and delegates to validateUsername (the profanity gate — see Settings & profile), and nextCookies() bridges Better Auth's cookie handling into Next.js server actions. A role additional field (default user, input: false so clients can't set it) is what web-shell's adminProcedure later checks. sendResetPassword is the one outbound hook — it renders resetPasswordEmail and ships it through the email gateway.

The HTTP surface is a two-line catch-all: app/api/auth/[...all]/route.ts re-exports GET/POST from toNextJsHandler(auth) — every credential operation (sign-up, sign-in, sign-out, reset, updateUser) is a Better Auth endpoint under /api/auth/*, none of it goes through tRPC. The tRPC side only reads: authRouter.me is a public procedure that projects ctx.session.user down to { id, email, name, username, displayUsername, role } and returns null when signed out — it is the session probe used by nearly every server page in the app.

Client side, lib/auth-client.ts is a 6-line createAuthClient with usernameClient(). Two tiny components round out the loop: SessionRefresh (mounted once in web-shell's root layout) just calls authClient.useSession() so an active tab keeps its session cookie fresh, and SignOutButton calls authClient.signOut() then pushes to /sign-in with a router refresh.

Password reset end to end: the only app code involved is the sendResetPassword hook and the HTML template — token issuance, verification, and credential update are all Better Auth.

Auth pages src/app/(auth)

↑ back to cards

The (auth) route group holds four client-form pages under a shared minimal layout ((auth)/layout.tsx — just a PPGL wordmark header, no app nav). Each page is the same shape: local useState form, one authClient call, inline error, no tRPC mutation. Sign-in and sign-up wrap their forms in Suspense because they read useSearchParams.

The interesting wrinkle is the invite handoff: both sign-in and sign-up read ?invite=<code>. Sign-in redirects to /league/join?code=… after success; sign-up goes further — after authClient.signUp.email succeeds it directly fires the league.join tRPC mutation (leagues) and lands the new user inside their league, falling back to /?joinError=1 if the join fails after the account was created. This is the one place the auth surface reaches into another domain's write API.

Sign-up submits username (lowercased) plus displayUsername (as typed) — the case-insensitive/display split that Better Auth's username plugin stores as two columns. All post-auth navigations use window.location.href rather than the router on purpose: a full navigation forces server components (the nav bar) to re-render with the new session cookie.

password-input.tsx is the shared show/hide password field (eye toggle, tabIndex={-1} so it doesn't trap keyboard flow); the settings page reuses it too. sign-in/layout.tsx and sign-up/layout.tsx exist only to attach metadata titles to client pages.

Settings & profile src/app/settings · src/server

↑ back to cards

/settings is a server page that gates on auth.me (redirect to /sign-in when null) and stacks three client sections. Two of them are Better Auth mutations: AccountSection re-validates the username client-side (length, charset, then validateUsername) before authClient.updateUser({ username, displayUsername }), and PasswordSection calls authClient.changePassword with current + new. Neither touches app-owned tables.

NotificationSection is the exception and the domain's only conventional stack: a toggle bound to user.getNotificationPreferences / user.updateNotificationPreferences, flowing userRouter → userService → userRepo, which reads and writes the single pickReminderEmail boolean on the user table. The service layer is a pass-through today — it exists to keep the router thin per the repo convention, and it's where profile logic would grow.

lib/profanity.ts is the shared username policy: a charset/length regex plus a substring check against the bad-words list. It runs in both places — client-side in AccountSection for fast feedback, and server-side inside the Better Auth username plugin's usernameValidator (see Better Auth core), so bypassing the form doesn't bypass the rule.

Pick-reminder notifications src/server/services · src/server/repositories

↑ back to cards

The pipeline that nags league members before a tournament locks. Both files deliberately drop the server-only guard and import db-core directly (the same pattern as the sync repos) because their caller is golf-data's cron route, not a tRPC procedure.

sendPickReminders(tournamentId, name) runs two eligibility queries: members with no pick for the tournament, and — only if the field is populated — members whose picked golfer has no CONFIRMED entry in tournamentField. Both queries exclude bots, exclude users who toggled pickReminderEmail off, restrict to ACTIVE members of PRIVATE leagues, and anti-join against notification_log so nobody is emailed twice. Results are aggregated per user across leagues (a pick-not-in-field finding upgrades a no-pick row for the same league), rendered with pickReminderEmail and shipped as one sendBatchEmail call.

Delivery logging is best-effort by design: after a successful batch, one notification_log row per user is inserted with ON CONFLICT DO NOTHING on (userId, tournamentId, type); a failed insert is logged but not fatal since the unique constraint dedups the next run. A known trade-off is annotated in the code: if a 100+ recipient batch fails on a later chunk, earlier chunks were sent but never logged, so those users could be re-emailed next tick.

sendTestPickReminder(tournamentId, email) is the admin-facing sibling: it rebuilds a real user's actual league/pick state so the preview email is truthful, but skips notification_log entirely so it can be re-sent freely. findUpcomingTournamentsWithinHours is the scheduling query the cron uses to decide which tournaments are within the reminder window.

One email per user per tournament, ever — the (userId, tournamentId, type) unique constraint is the idempotency key, not the cron schedule.

Transactional email src/server/services

↑ back to cards

email.ts is the only file that talks to Resend: a lazily-constructed singleton client (so builds without the API key don't explode at import time), a configurable EMAIL_FROM sender with a dev fallback, and two functions — sendEmail for one message and sendBatchEmail, which chunks into groups of 100 to respect Resend's batch-API ceiling and throws on the first failed chunk.

email-templates.ts keeps rendering separate from delivery: pure functions returning { subject, html } built from a shared table-layout layout() (inline styles only — email-client-safe, PPGL green header, a footer link to /settings as the manage-notifications escape hatch) and a button() helper. resetPasswordEmail takes the Better Auth reset link; pickReminderEmail renders the per-league issue list ("No pick yet" vs "golfer is not in the field") with a dashboard CTA. Links are absolute via NEXT_PUBLIC_APP_URL.

Two producers, one gateway: the Better Auth sendResetPassword hook and the notification service. Nothing else in the repo sends mail.

Data it owns

Five Prisma models in prisma/schema.prisma. The first four are Better Auth's tables — the library reads and writes them directly through the shared Kysely instance; app code only ever touches the extra columns it added to User.

UserIdentity plus PPGL extensions: username/displayUsername (case-insensitive vs as-typed), role (checked by web-shell's adminProcedure), isBot (excluded from reminders; used by game's bot picks), pickReminderEmail (the settings toggle). Referenced by nearly every domain via LeagueMember, commissioner relations, and adjustment audit fields.
AccountPer-provider credential row — for email/password this is where the hashed password lives, cascade-deleted with the user.
SessionServer-side session rows keyed by unique token, with IP/user-agent metadata; the cookie references these.
VerificationShort-lived verification values (password-reset tokens) keyed by identifier, with expiry.
NotificationLogSend-dedup ledger: unique (userId, tournamentId, type) — one PICK_REMINDER per user per tournament, ever.

User is this domain's biggest export in disguise: every other domain's rows ultimately hang off user.id, but only accounts (and Better Auth) may write the identity columns.

Seams

A small tRPC surface (identity is read-mostly over tRPC), one HTTP catch-all that is the credential API, and two function-level seams consumed by cron and admin.

↑ Exposes
  • auth.me — Public session probe — `null` or the projected user; called by server pages across every UI domain
  • user.getNotificationPreferences — Protected — reads the `pickReminderEmail` flag
  • user.updateNotificationPreferences — Protected — the settings toggle write
  • /api/auth/[...all] — Better Auth catch-all (`GET`/`POST`) — sign-up, sign-in, sign-out, password reset/change, `updateUser` all live here, outside tRPC
↓ Depends on
web-shell srcleagues src/server/routers/league.ts

Module-level seams beyond tRPC: auth itself is imported by web-shell's trpc.ts (getSession per request) and admin's service (requestPasswordReset); golf-data's cron-sync imports sendPickReminders + findUpcomingTournamentsWithinHours; admin's router imports sendTestPickReminder. Dependencies going out: web-shell supplies router/protectedProcedure, db/db-core, and the tRPC clients; the sign-up page calls leagues' league.join. The notification queries also read pick, league, and tournament tables owned by game, leagues, and golf-data — data-level reads with no code import.
