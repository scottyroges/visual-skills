Earnings & standings

What it isThe money engine: turns LOCKED picks × TournamentResult into season totals, layers in commissioner adjustments, and serves every leaderboard and standings view.
Pathsrc/server (earnings + leaderboard stack) · src/components (standings UI)
Files12
Entry pointsleaderboard and earningsAdjustment tRPC routers; rollupForTournament / recalculateSeasonForMember called directly by golf-data sync, pick service, and admin.
GuaranteeThe rollup is idempotent — every trigger recomputes a member's season from scratch, so re-running it is always safe (seed:rollup exists for exactly this).
Big idea
LeagueMemberSeason is a materialized rollup, not a ledger.

Nothing ever increments the stored totals. Every trigger — tournament finalize, commissioner pick edit, adjustment add/delete — recomputes the member's whole season with one SQL aggregate over LOCKED picks joined to results, adds the adjustment sum, and overwrites the row. Reads (leaderboards, standings) then never touch the pick table for season totals — they do one indexed read of the cache.

The rollup flow

Every path into the domain converges on recalculateSeasonForMember: two SQL aggregates, one overwrite of the cache row, and the leaderboards read only the cache.

Blue nodes are the five triggers (three live in other domains). The recompute counts only pick.status = 'LOCKED' rows on non-excluded COMPLETED/FINALIZING tournaments.

☉The UPDATE never inserts. updateMemberSeasonStats assumes the LeagueMemberSeason row already exists — leagues creates it when a member joins. If the row is missing, the rollup logs [earnings-rollup] No LeagueMemberSeason row found … and skips the member rather than failing the whole rollup.

In depth

Rollup core src/server/services/earnings-core.ts

↑ back to cards

Three functions, one shape: rollupForTournament(tournamentId) finds every member with a LOCKED pick on that tournament (findAffectedMembers) and recomputes each; recalculateSeasonForLeague does the same for every member of a league; both funnel into recalculateSeasonForMember, the single write path. It runs calculatePickBasedStats (one SQL aggregate over LOCKED picks joined to tournamentResult, restricted to season, status in (COMPLETED, FINALIZING), excluded = false), adds sumByLeagueMember from the adjustments repo, and overwrites the LeagueMemberSeason row's totalEarnings / totalPicks / totalMissedCuts.

Missed cuts are a SQL heuristic, not a stored flag: a result row that exists with earnings = 0 and finish_position null-or-zero counts as a missed cut — no result row yet means the tournament simply hasn't produced one.

The server-only split is the subtle part: earnings-core.ts and its repositories import @/lib/db-core and carry no server-only guard, so prisma/seed/rollup.ts (the seed:rollup npm script) can execute the exact same code from the CLI. earnings.ts is a three-line server-only re-export shim — the version app code is supposed to import. Other server services (sync, pick, admin) import earnings-core directly.

Commissioner adjustments src/server/services/earnings-adjustment.ts

↑ back to cards

A tiny append/delete ledger for league-specific money that ESPN doesn't know about — side bets, penalties. resolveCommissioner gates every write: the target member and league must exist, the league must not be the GLOBAL league (adjustments are FORBIDDEN there), and the caller must be league.commissionerId. Reads are looser on purpose — getAdjustments lets any member of the league view them, for transparency.

Writes are transactional and audited: addAdjustment / deleteAdjustment wrap the earningsAdjustment insert/delete together with a CommissionerAction audit row (ADD_ADJUSTMENT / DELETE_ADJUSTMENT, details JSON-serialized) in one withTransaction, then trigger recalculateSeasonForMember outside the transaction so the stored totals absorb the change.

The router validates hard bounds — non-zero integer amount within ±999,999,999, memo 1–100 chars. EarningsAdjustments (client component) renders the list with signed green/red amounts and, for the commissioner, an inline add form and a confirm-to-delete flow; it calls router.refresh() on success so the server-rendered totals re-fetch.

Leaderboard queries src/server/services/leaderboard.ts

↑ back to cards

The read side never aggregates picks for season totals — findStandings is a straight join of leagueMemberSeason × leagueMember (ACTIVE only) × user, ordered by totalEarnings desc with a stable secondary sort on member id (a TODO in the repo admits there's no real tiebreaker yet). getGlobalStandings (public, for the home page) and getLeagueStandings (membership-gated) share the same four-query Promise.all: standings page, total count, projected earnings, and the leader's max for diffFromLeader.

Projected earnings are display-only. findProjectedEarnings sums results from IN_PROGRESS tournaments per member — live money that is deliberately not in totalEarnings and never persisted; it decorates each standings row so the UI can show what's pending this week.

Weekly standings invert the join direction: findWeeklyStandings starts from leagueMember and LEFT JOINs the LOCKED pick for one tournament, so members without a pick still appear; the service splits rows into picks (ranked by earnings, then score-to-par) and membersWithoutPicks. memberStanding / memberStandingByUser are the single-row lookups behind the member season pages, both verifying the caller shares the league first.

Standings surfaces src/components/league/weekly-standings-table.tsx

↑ back to cards

WeeklyStandingsTable is the one shared standings renderer: ranked pick rows (member link, golfer avatar + link, live finish position, color-coded score-to-par, earnings with an est. tag when isLive) followed by dimmed no-pick rows — mirroring the picks / membersWithoutPicks split the service returns. The caller's own row is highlighted via currentUserId. It collapses responsively: on mobile the golfer moves under the member name.

/member/[leagueId]/[memberId] is a legacy server page kept only as a redirect: it resolves memberId → userId through leaderboard.memberStanding (setting the active-league cookie on the way) and forwards to /season/[userId], falling back to /season if the lookup fails.

Data it owns

Two tables: the materialized season cache and the manual adjustment ledger.

LeagueMemberSeasonOne row per member per season (@@unique([leagueMemberId, season])): totalEarnings, totalPicks, totalMissedCuts. Write-owned here, row-created elsewhere — leagues inserts the row when a member joins; this domain only UPDATEs it.
EarningsAdjustmentCommissioner ledger entries: signed integer amount, required memo, createdById (relation earningsAdjustmentsCreated on User). Summed into totalEarnings on every recompute.

The raw money inputs — Pick and TournamentResult — belong to game and golf-data; this domain reads them but never writes them. CommissionerAction audit rows written by adjustments belong to leagues.

Seams

Two tRPC routers outward, three service functions callable by neighbor services, three domains leaned on.

↑ Exposes
  • leaderboard.global — **public** — season standings for the global league; the signed-out home page leaderboard.
  • leaderboard.byLeague — Paginated league standings with rank, projected earnings, `diffFromLeader`.
  • leaderboard.weeklyStandings — One tournament's picks + no-pick members for a league; league and tournament pages.
  • leaderboard.memberStanding — Single member's season row by memberId (also the legacy `/member` redirect's lookup).
  • leaderboard.memberStandingByUser — Same, keyed by userId — what `/season/[userId]` uses.
  • earningsAdjustment.add — Commissioner-only; blocked on the GLOBAL league; audited; triggers recompute.
  • earningsAdjustment.delete — Commissioner-only; audited; triggers recompute.
  • earningsAdjustment.list — Any league member may read (transparency).
