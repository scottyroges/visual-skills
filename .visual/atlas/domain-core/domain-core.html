Domain

The shared block→HTML rendering engine every generator builds on — a typed Block vocabulary, per-type renderers and helpers, and an assemble() that turns a Block[] into one self-contained HTML document.

Pathsrc (the engine modules) + src/renderers/* (one renderer per block type).
Files15 — blocks.ts, assemble.ts, html.ts, highlight.ts, lint-blocks.ts, version.ts, and nine renderers.
RoleOwns the Block union and the shared render primitives. doc feeds its Block[] straight to assemble(); recap, spec, and atlas ship their own shells that reuse core's renderers, markdown, and escaping.
The load-bearing idea
"One Block vocabulary and one set of render primitives — every product's page is built from the same core pieces."

A product's job is to build blocks — diagrams, diffs, prose, file trees, annotated code. The plain-doc path is assemble(): it walks the Block[], dispatches each block to its renderer by a switch (b.type), inlines the CSS + viewer JS + dark-mode toggle assets, and returns one portable HTML string. The recap, spec, and atlas shells live in their own domains but build on the same union, renderers, markdown, and escaping — add a primitive once here and every generator can use it.

How a document is assembled

architecture

The doc product hands assemble() a Block[] (recap, spec, and atlas run their own shells on the same pieces). block-model src/blocks.ts

↑ back to cards

Block is a discriminated union keyed on type — diagram, schema, api, file-tree, diff, prose, annotated-code, questions, group, tabs, overview. This is the entire contract between a product and the engine: a generator never touches HTML, it only constructs typed blocks. assemble's exhaustive switch (b.type) is checked against this union, so adding a variant is a compile error until a renderer handles it.

Two variants are containers that nest one level deep: group (a labelled <details> of child blocks) and tabs (a CSS-only switcher, max 6 tabs). diff and overview may each carry an optional embedded diagram. The only logic here is isDiagramBlock, the type guard assemble uses to find every diagram/schema block — including those nested inside groups, tabs, diffs, and overviews — so they can be rendered through the diagrams domain up front.

assemble + shell src/assemble.ts · src/html.ts

↑ back to cards

assemble(blocks, opts) is the plain-document pipeline — the doc generator calls it directly, while the recap/spec/atlas shells reuse its pieces rather than the function itself. It first asserts every block id is unique recursively (anchors and in-page #cross-links depend on it), runs lintBlocks through opts.onWarn, then collects every diagram/schema block — including nested ones — and renders them up front through the diagrams domain, keyed by id. It then maps each top-level block to HTML via renderBlock, a switch (b.type) that calls the matching renderer; group and tabs recurse. Finally it inlines assets/template.css and assets/viewer.js, plus the dark-mode toggle assets — assets/theme-head.js (a synchronous <head> script that sets data-theme on <html> from localStorage["vs-theme"]/OS preference before paint), assets/theme-toggle.js (injects the .vs-theme-toggle button and wires click + OS-change), and assets/theme.css (appended after template.css in the <style>) — stamps an optional <meta generator> and footer, and returns a single self-contained HTML string — no external assets, by design.

html.ts is the tiny shared safety layer: escapeHtml (used by every renderer for any text that reaches HTML) and SAFE_HREF, the regex that allows only #fragment and http(s):// links so a javascript:/data: URL can't slip through the overview and review link renderers. assemble also injects each block's id as an anchor on its <section> so the cross-links resolve.

renderers src/renderers/*.ts

↑ back to cards

Nine modules: seven per-block-type functions — renderProse, renderDiff, renderApi, renderFileTree, renderAnnotatedCode, renderOverview, renderQuestions — plus renderLegend (the diagram color key) and the shared renderMarkdown. Each block renderer takes its block (some plus an onWarn or a pre-rendered extra) and returns an HTML fragment wrapped in a vs-block section; assemble adds the anchor and stitches them together. There is no shared base class — the union + the dispatch switch are the only structure.

markdown.ts is the substrate: renderMarkdown parses GitHub-flavored markdown, highlights fenced code via the highlight module, and runs the result through sanitize-html with a strict tag/attribute allowlist; renderInlineMarkdown does the one-line variant for headlines and bullets. prose, diff, and overview all build on it. diff and annotated-code use highlightLines to align highlighted code to a gutter; file-tree collapses single-child directory chains and links filenames back to their diff blocks; legend paints the auto-derived diagram color key.

highlight src/highlight.ts

↑ back to cards

A thin wrapper over Shiki (dual github-light/github-dark themes with defaultColor: "light" — inline color: stays light-only, the dark palette rides along in --shiki-dark/--shiki-dark-bg custom properties that theme.css flips on for the dark-mode toggle — plus a fixed set of loaded languages) behind a lazy promise singleton, so the highlighter is created at most once per process. highlightCode returns a complete Shiki <pre> for fenced markdown code; highlightLines returns the inner HTML of each source line so diff and annotated-code can interleave code with gutters and per-line notes. langFromPath maps a file extension to a Shiki language id.

Every path is defensive: an unloaded language, a Shiki error, or a line-count mismatch degrades to escaped plaintext (and an onWarn note) rather than throwing — a highlighting failure must never break a document.

lint src/lint-blocks.ts

↑ back to cards

lintBlocks(blocks) returns authoring warnings in document order — it never throws and never blocks rendering. assemble pipes them to opts.onWarn. The guards nudge quality: a group with no description, a diff description that is a long single paragraph (a wall of text), a diagram that applies no semantic color, an editable-eligible diagram kind that dropped its mermaid source (silently losing the Excalidraw upgrade), and too-many / oversize-edge diagrams overall.

It walks the same nested structure assemble does — groups, tabs, embedded diff/overview diagrams — using the diagram-color helpers to detect which roles a d2/mermaid source applies. The intent is to catch the mistakes that make a document read worse without making the build fail.

Key files
lint-blocks.tsThe group / diff-prose / diagram authoring guards, recursing through nested blocks.
Key exports
lintBlocks()→ string[] of non-blocking authoring warnings in document order.
Connections

version src/version.ts

↑ back to cards

generatorStamp() produces the provenance string visual-skills@<git-short-sha>[-dirty] · <ISO timestamp>. It shells out to git for the short SHA and a porcelain status (the -dirty suffix when the tree has uncommitted changes); if git is unavailable it falls back to the package.json version, then to "unknown". The recap and doc CLIs pass the result as opts.generator, which the shell writes into the page <meta generator> and footer so any saved document records exactly which build produced it (the spec and atlas CLIs default to a static skill label instead).

The block model it owns

Core owns the Block discriminated union — the type model every product builds against and the engine renders. These are the variants it defines.

BlockThe discriminated union itself — the contract between products and assemble().
DiagramBlock · SchemaBlockA d2 (+ optional mermaid) diagram, and the d2-only ERD schema variant.
ApiBlockA tRPC procedure table — name, kind, auth, input, and change marker per row.
DiffBlockA file diff: path, optional markdown description, optional embedded diagram, and hunks.
FileTreeBlockA change tree of FileChange entries, linkable back to their diff blocks.
ProseBlock · OverviewBlockA markdown prose section, and the TL;DR headline + key points (optional lead diagram, plus optional review facets/risk).
AnnotatedCodeBlockA code listing with per-line margin notes, gutter-aligned to highlighted source.
QuestionsBlockOpen-question cards, each with a recommended default.
GroupBlock · TabsBlockThe two one-level-deep containers: a labelled <details> group and a CSS-only tab switcher.

All HTML is produced at render time and inlined — core owns no persisted data, only the in-memory block types and the document string assemble() returns.

Seams

A small typed surface: products build blocks against the Block union and render through core's primitives; core in turn calls the diagrams domain to render every diagram up front.
