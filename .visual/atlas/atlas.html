Start here

A toolkit of four CLI generators that turn code, diffs, and specs into self-contained, visually-readable HTML — all built on one shared block→HTML rendering engine.

WhatFour generators ship as bin/ CLIs: visual-recap (PR / commit / diff review), visual-atlas (this codebase-map), visual-spec (design-spec / RFC page), and visual-doc (general illustrated doc). Each emits one standalone .html that opens over file:// — no server, no reader-side build.
ShapeLogic lives in src/ and is consumed by thin entrypoints in bin/. A rendering core turns a typed Block[] into a page; diagrams and code-intelligence are shared services the products lean on. Two products scan a real repo (recap, atlas); two are pure authoring (spec, doc).
SpineEvery product is the same pipeline: build a typed Block[] → assemble() walks it, calling one renderer per block type → rendered diagrams and highlighted code get inlined → one self-contained HTML file.
1
One Block union, four products.
There is no per-product renderer. Each CLI just assembles a different array of the same Block types. Read src/blocks.ts then src/assemble.ts and you have the spine of all four generators.
2
The output is a single self-contained file.
Every generator inlines its CSS, syntax-highlighted code, and rendered diagrams into one .html that works over file://. Self-containment is a hard product constraint, not a nicety — it shapes how diagrams and assets are handled.
3
Two products read code; two only author.
recap and atlas understand a real repo — git diffs, Prisma/tRPC parsing, import graphs — through the code-intel layer. spec and doc take hand-authored blocks JSON and never scan. That split explains why code-intel exists.
4
d2 is the diagram floor; the atlas is hybrid.
Diagrams compile through the d2 CLI to static SVG (with an optional editable-Excalidraw upgrade), and one palette colors every diagram. The atlas itself is hybrid: a mechanical scan drafts JSON, humans enrich the prose, and the committed JSON re-renders with no scan.

The domain map

layered

Six domains in three tiers. A foundation (diagrams) and an engine (core) sit at the bottom; an intelligence layer (code-intel) reads real repos; three surface products stack on top. Arrows are depends on — and the two scanning products reuse recap's review shell.

Central engine (every product uses it)

Read bottom-up: diagrams and core are shared infrastructure; code-intel reads real repos for the scanning products; recap / atlas / authoring are the user-facing CLIs. core is highlighted because all four products route through its assemble().

Spine

runtime

The one flow worth memorizing — it is identical across all four generators. A product builds a typed Block[]; assemble() walks it and dispatches one renderer per block type; diagram and code blocks get compiled and inlined; the result is a single self-contained HTML file.

What differs between products is only the left edge — how the Block[] is gathered (a git diff, a repo scan, or hand-authored JSON). From assemble() rightward, the path is shared.

The 6 domains

all with pages

Each domain owns one slice of the toolkit. The bottom two are shared infrastructure; code-intel is the repo-reading layer; the top three are the user-facing products. Every tile links to a full deep-dive page.

coresrc/Engine

The shared block→HTML rendering engine. Defines the Block union every product targets, walks it with assemble(), dispatches one renderer per block type, highlights code, lints, and stamps the output. There is no per-product render path — only this one.
