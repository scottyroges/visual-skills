Start here

A toolkit of four CLI generators that turn code, diffs, and specs into self-contained, visually-readable HTML — all built on one shared block→HTML rendering engine.

WhatFour generators ship as bin/ CLIs: visual-recap (PR / commit / diff review), visual-atlas (this codebase-map), visual-spec (design-spec / RFC page), and visual-doc (general illustrated doc). Each emits one standalone .html that opens over file:// — no server, no reader-side build.
ShapeLogic lives in src/ and is consumed by thin entrypoints in bin/. A rendering core turns a typed Block[] into a page; diagrams and code-intelligence are shared services the products lean on. Two products scan a real repo (recap, atlas); three are pure authoring (spec, doc, quiz).
SpineEvery product is the same pipeline: build a typed Block[] → its assembler walks it, calling one renderer per block type → rendered diagrams and highlighted code get inlined → one self-contained HTML file.
1
One pattern, four products.
Each product pairs a block vocabulary with a thin assembler built on core's shared primitives — the renderers, lint, and highlighting. Only the doc CLI calls core's assemble() directly; recap, spec, and atlas each have their own assembler over the same parts. Read src/blocks.ts then src/assemble.ts and you have the pattern the other three follow.
2
The output is a single self-contained file.
Every generator inlines its CSS, syntax-highlighted code, and rendered diagrams into one .html that works over file://. Self-containment is a hard product constraint, not a nicety — it shapes how diagrams and assets are handled.
3
Two products read code; two only author.
recap and atlas understand a real repo — git diffs, Prisma/tRPC parsing, import graphs — through the code-intel layer. spec and doc take hand-authored blocks JSON and never scan. That split explains why code-intel exists.
4
d2 is the diagram floor; the atlas is hybrid.
Diagrams compile through the d2 CLI to static SVG (with an optional editable-Excalidraw upgrade), and one palette colors every diagram. The atlas itself is hybrid: a mechanical scan drafts JSON, humans enrich the prose, and the committed JSON re-renders with no scan.

The domain map

layered

Six domains in three tiers. A foundation (diagrams) and an engine (core) sit at the bottom; an intelligence layer (code-intel) reads real repos; three surface products stack on top. Arrows are depends on — and atlas and authoring share recap's review shell (its css, viewer script, and section helpers).

Central engine (every product uses it)

Read bottom-up: diagrams and core are shared infrastructure; code-intel reads real repos for the scanning products; recap / atlas / authoring are the user-facing CLIs. core is highlighted because all five products build on its Block primitives and renderers.

Spine

runtime

The one flow worth memorizing — it is identical across all four generators. A product builds a typed Block[]; its assembler walks it and dispatches one renderer per block type; diagram and code blocks get compiled and inlined; the result is a single self-contained HTML file.

What differs between products is only the left edge — how the Block[] is gathered (a git diff, a repo scan, or hand-authored JSON). From the assembler rightward, the parts are shared: core's renderers, highlight, and lint.

The 6 domains

all with pages

Each domain owns one slice of the toolkit. The bottom two are shared infrastructure; code-intel is the repo-reading layer; the top three are the user-facing products. Every tile links to a full deep-dive page.

coresrc/Engine

The shared block→HTML rendering engine. Defines the Block union and the renderers every product draws on, highlights code, lints, and stamps the output. assemble() is the doc CLI's assembler; the other products' assemblers reuse core's renderers, lint, and highlighting.
