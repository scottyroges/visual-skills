Domain

The visual-atlas product — it scans a repo, groups modules into domains, drafts a two-level map, and renders atlas.html plus a page per domain.

OwnsThe atlas block model (the AtlasBlock union — atlas-tldr, domain-map, domain-index, domain-tldr, components, depth, owns, seams, diagram-section) and the atlas.domains.json grouping config.
WhyA standing onboarding map: the atlas page gives the all-domains picture, each domain-<slug>/ page is a deep-dive + reference. Mechanical scan handles the inventory; a human writes the prose and the why.
Self-describingThe page you are reading is this domain. The very generator documented below produced it — the scanner drafted these cards, then a human enriched them.
The load-bearing idea
"Hybrid — the scanner does the inventory, the human writes the domains and the why."

scanInventory finds modules, imports, and exports; firstGuessConfig/reconcile group them into domains from atlas.domains.json globs and report drift (never auto-applies it); buildAtlasDraft/buildDomainDraft emit a skeleton with empty purpose/detail fields. The draft is written only when absent (never clobbers authored prose unless --force), and lint-atlas flags the blank fields a human still needs to fill.

CLI bin/atlas.ts

↑ back to cards

The single entry point with three modes (paths may be relative — they resolve against the cwd). Full scan (--repo <dir> --out <dir>) loads or first-guesses atlas.domains.json, scans the repo, reconciles + reports drift, writes the config, drafts atlas.json and each domain-<slug>/domain-<slug>.json only when absent (or always with --force), then renders every doc. Single-domain (--domain <slug>) rescans and regenerates just that page (and prints a tile-update note). Render-only replays committed JSON — --all <dir> re-renders the atlas plus every domain-<slug>/ folder, --blocks <file> renders one page.

writeDraftIfAbsent is the no-clobber guarantee: it skips any existing draft unless --force. The layout puts atlas.{html,json} + atlas.domains.json at the top of the out dir, and each domain self-contained in its own domain-<slug>/ folder (page + diagram sidecars). It also warns about orphaned domain-*/ folders left after a regroup — but never deletes files. Every mode except --blocks also refreshes atlas-check.mjs in the out dir (emitChecker copies the tool-owned assets/atlas-check.mjs, always overwritten) — the self-contained drift checker target repos commit and run from pre-commit/CI with plain Node.

config + drift src/atlas-config.ts

↑ back to cards

atlas.domains.json is the human-owned source of truth: each DomainConfig carries a slug, name, and globs (the editable lever); the scanner fills the resolved modules. firstGuessConfig seeds a fresh config — one domain per immediate child directory of each srcRoot — so a new repo starts grouped, not blank. reconcile refills each domain's modules from its globs against the live inventory while preserving every human-owned field.

Crucially, reconcile reports drift but never applies it: newModules (in the repo, matched by no glob), stalePaths (config paths no longer present), and emptyDomains (globs resolving to nothing). The human resolves drift by editing globs. matchGlob is the tiny anchored matcher — ** spans segments, * spans within one.

scan + draft src/gather-atlas.ts

↑ back to cards

The hub of the full-scan flow. scanInventory walks the srcRoots, and per module records resolved in-repo import keys, exported names, and whether it is a tRPC router — skipping generated/test trees and co-located *.test/*.spec files. It also collects Prisma model names. aggregateDomainEdges rolls module→module imports up to cross-domain slug→slug edges (intra-domain edges dropped).

buildAtlasDraft emits the atlas-page doc: a tile per domain (with file counts and dep edges) plus the mechanical domainMapDiagram. buildDomainDraft groups a domain's modules by subdirectory into component cards + deep sections, listing real files and exports — but leaves every purpose/detail field blank for the human. The scanner never writes prose.

block model src/atlas-blocks.ts

↑ back to cards

The type backbone of every atlas page — and the file this very section documents. The AtlasBlock union has nine members: atlas-page blocks (atlas-tldr, domain-map, domain-index), domain-page blocks (domain-tldr, components, depth, owns, seams), and the shared diagram-section. ComponentDeep is one expandable depth section; AtlasDiagram is a rendered d2/mermaid pair.

Beyond the types it carries page-option shapes (AtlasOpts/DomainOpts), LAYER_DOTS (the layer → fill;stroke color table), and small helpers: collectAtlasDiagrams pulls every rendered diagram in document order, assertUniqueAtlasIds enforces unique block/component/diagram ids, and isAtlasChapter/atlasChapterLabel drive the numbered chapter nav.

export type AtlasBlock =
  | AtlasTldrBlock | DomainMapBlock | DomainIndexBlock
  | DomainTldrBlock | ComponentsBlock | DepthBlock
  | OwnsBlock | SeamsBlock
  | DiagramSectionBlock;

render src/assemble-atlas.ts

↑ back to cards

Turns a block list into a self-contained HTML page. assembleAtlas renders the map page (lead, domain-map, tile index); assembleDomain renders a domain page (lead, components, internal-arch, depth, owns, seams). renderAtlasBlock dispatches per block type, renderAtlasDiagram emits each diagram's d2/mermaid render with its authored legend, and atlasLegend builds the legend rows.

Every diagram is rendered through the diagrams domain up front (collected via collectAtlasDiagrams), then looked up by id while rendering. The output reuses the review app-shell/CSS from the recap domain — including the same inlined dark-mode toggle (theme-head.js head-apply script, theme-toggle.js button, theme.css appended last in <style>) — so atlas pages share the toolkit's look and theme. The CLI calls these two functions and writes the HTML next to the page JSON.

completeness lint src/lint-atlas.ts

↑ back to cards

The demo-standard floor, mirroring the spec/recap linters: heuristics surfaced via onWarn, never hard errors. lintAtlas checks the atlas page leads with a atlas-tldr 'Start here', shows a domain map, and has a domain-index whose tiles all have a real purpose — flagging any blank ones the scanner left.

lintDomain checks a domain page opens with a domain-tldr, lists its components, includes an internal-arch diagram-section when it has four-plus deep components (LARGE_COMPONENTS), and names its seams. The warnings tell the human exactly which scanner-drafted fields still need enriching — closing the hybrid loop.

Data it owns

Atlas owns the page block model and the grouping config — the two contracts every atlas page is built from. The block types below are the shape of a page; AtlasConfig/atlas.domains.json is the human-owned grouping.

AtlasConfig · atlas.domains.jsonThe human-owned grouping: domains as slug/name/globs, with scanner-resolved modules. The one lever for which files form which domain.
atlas-tldr · domain-map · domain-indexThe atlas-page blocks: the 'Start here' lead, the all-domains map, and the grid of domain tiles.
domain-tldr · components · depth · owns · seamsThe domain-page blocks: the lead, the component cards, the in-depth sections, the data-owned table (this block), and the neighbor seams.
diagram-sectionThe shared rendered-diagram block — the atlas spine/map and a domain page's internal-arch.
Inventory · ModuleInfo · DriftThe mechanical scan products: per-module imports/exports/router flags, Prisma models, and the reported config drift.

The block model lives in atlas-blocks.ts; the config types in atlas-config.ts. The scanner fills modules and the draft skeletons; a human owns globs, purpose, and detail.

Seams

Exposes the atlas CLI and its block model; depends on the rest of the toolkit for parsing, rendering, diagrams, and the shared app-shell.
