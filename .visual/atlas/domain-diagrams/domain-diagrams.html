How a diagram renders

architecture

A diagram source is either native d2 or mermaid. renderDiagram is the hub: it prepends the palette's D2_CLASS_PRELUDE, shells out to the external d2 CLI for the SVG floor, and — only when the caller opts in, for eligible flowchart-class kinds — upgrades the mermaid source to an editable Excalidraw scene. mermaidFlowchartToD2 is the converter branch that gives a mermaid-authored diagram its d2 floor.

renderDiagram is the hub; the d2 CLI is the only external dependency. The Excalidraw upgrade is opt-in and falls back to the d2 SVG on any error.

render-diagram src/render-diagram.ts

↑ back to cards

renderDiagram(block, opts, deps) is the heart of the domain. It first compiles the block's d2 source to an SVG by writing D2_CLASS_PRELUDE + the source to a temp file and shelling out to the d2 binary (d2 --theme 0 --pad 24). This is the dependable floor: it renders every diagram kind with no browser. If d2 is missing or the source is invalid, the call degrades to a visible placeholder SVG (a warning box) rather than throwing — so one bad diagram never breaks a whole document.

The second stage is an opt-in upgrade — off by default: a missing or false excalidraw option stays on the d2 floor. Routing is by diagram kind, not by toolchain availability: only flowchart, architecture, sequence, and class kinds are eligible, because those are the ones mermaid-to-excalidraw converts to native editable elements (ERD and stateDiagram silently rasterize, which is worse than the d2 sketch). When the caller explicitly opts in (excalidraw: true), the kind is eligible, and the Excalidraw toolchain is present, it converts the mermaid source to an editable .excalidraw scene plus a display SVG, forcing all text to dark INK for legibility. Any failure falls back to the d2 SVG already in hand — same aesthetic, never broken. renderAll maps this over many blocks in order.

palette src/diagram-colors.ts

↑ back to cards

PALETTE is the single source of truth for the six semantic roles (changed, added, removed, actor, external, store), each a { fill, stroke, bold? }. From this one object two representations are generated so they can never drift: D2_CLASS_PRELUDE (the d2 classes {} block that render-diagram injects before every render, letting any source apply class: <role>) and MERMAID_CLASSDEFS (the matching classDef lines an editable flowchart includes so its Excalidraw scene stays colored). INK is the dark label color that guarantees readable contrast on every fill.

The legend logic keeps the rendered legend honest. rolesInSource scans a source for roles that are applied (class: role in d2, :::role or class a,b role; in mermaid) — excluding mere classDef definitions — and legendRolesForRender keys off the actual renderer: the d2 SVG carries d2 class: colors, but an Excalidraw scene carries only the mermaid classDefs, so a mermaid sequence (which has no class mechanism) renders colorless and correctly shows no legend even though its d2 floor defined roles.

mermaid-to-d2 src/mermaid-to-d2.ts

↑ back to cards

mermaidFlowchartToD2(mermaid) translates the common mermaid flowchart subset into equivalent d2, so a mermaid-authored diagram still has the dependable d2 render floor. It parses the graph/flowchart header for direction, walks each edge line splitting on the mermaid arrow forms (-->, -.->, ---, ==>), captures bracketed node labels and pipe edge labels, and emits direction:, node, and edge lines with everything quoted for safety.

Crucially it returns null for anything outside the subset (a missing header, an unparseable node shape) rather than emitting a half-broken diagram — the caller is expected to leave such a fence as inline code. That null-or-clean contract is what makes promote-mermaid safe to run over arbitrary authored prose.

promote-mermaid src/promote-mermaid.ts

↑ back to cards

promoteMermaidFences(blocks) lets authored markdown carry diagrams for free. It walks prose blocks and, for each fenced ```mermaid block, calls mermaidFlowchartToD2. When conversion succeeds it splits the prose, emitting the surrounding text as prose blocks and the fence as a real diagram block (kind flowchart, carrying both the converted d2 floor and the original mermaid source for the editable upgrade). When conversion returns null the fence is left untouched inline, so nothing is ever rendered half-broken.

This is the bridge from human-written markdown into the same render pipeline the structured generators use — the same DiagramBlock shape render-diagram consumes, so a hand-written mermaid diagram and a programmatically authored one render identically.

Data it owns

Diagrams owns the semantic role vocabulary — the six color roles every diagram across the toolkit speaks. PALETTE is the single source; the d2 prelude and mermaid classDefs are generated from it.

changedModified / the subject — bold amber (#ffd43b/#f08c00). The one role drawn with extra stroke weight so the subject pops.
addedNew — green (#d3f9d8/#37b24d).
removedDeleted — red (#ffe3e3/#f03e3e).
actorUser / initiator — blue (#d0ebff/#4dabf7).
externalThird-party system — gray (#f1f3f5/#adb5bd). The d2 CLI itself is tagged this in the arch diagram.
storeDatastore — db / cache / queue — violet (#e5dbff/#9775fa).

Roles are applied in sources (class: <role> in d2, class X <role>; in mermaid), never redefined — sources must not author their own top-level classes: block, since d2 silently merges it and would let a recipe shadow the canonical palette.
