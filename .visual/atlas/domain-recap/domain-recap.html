Domain

Turn a git target into a self-contained HTML review — and own the review shell every other product page reuses.

Pathbin/recap.ts (CLI) · src/*.ts (pipeline) · src/review/*.ts (the app-shell renderers)
Files19 — one CLI entry, eight pipeline modules, ten review-shell renderers.
RoleThe visual-recap product: resolve a PR / commit / branch / working-tree into a diff, parse and rank it, fold it into a Block[], and render the standing review document. The shell it renders is shared infrastructure — atlas and authoring assemble their pages on top of the same topbar / sidebar / sections.
The load-bearing idea
"recap is a pipeline, and its renderer is everyone's renderer."

Target → git.ts → parse-diff → diff-order → gather-recap → assemble-review → recap.html. The gather half is mechanical and TS/JS-aware; the render half (assemble-review + src/review/*) is the app-shell — topbar, sidebar, TL;DR, files-table, walkthrough, diagrams — that assemble-atlas and assemble-spec reuse to draw their pages. Enrich the gathered blocks.json, re-render with recap --blocks, and nothing in the pipeline needs to change.

Internal architecture

pipeline

gatherRecap is the spine. A Target resolves through git.ts to a Scope (base/head + raw diff); parse-diff and diff-order turn the diff into ranked DiffBlocks; buildBlocks folds them — plus a summary, where-it-fits, and api-surface diagrams — into a Block[]; assembleReview renders that through the src/review/* shell to recap.html. The shell is a shared seam: assemble-atlas and assemble-spec render their pages through it too.

The pieces, in depth

A full section per piece — what it does, how it works, its files and exports, and how it connects to the rest of the domain.

CLI bin/recap.ts

↑ back to cards

bin/recap.ts is the whole command-line surface. parseArgs reads --pr / --commit / --branch (--base to override) / default --working, which parseTarget folds into a Target discriminated union. The normal path calls gatherRecap to get { scope, blocks, adapter }, then assembleReview to produce recap.html. Output is written into a directory (a trailing .html is stripped): the HTML, any .excalidraw sidecars, and a sibling blocks.json all live together so the recap is self-contained.

The --blocks flag is the enrichment round-trip: it loads an already-gathered (and hand-enriched) blocks.json and renders it straight through assembleReview, skipping git entirely. That separation — gather once, enrich the JSON, re-render — is what lets the visual-recap skill add overviews, annotations, and groupings without re-running the diff. --emit-blocks writes the raw Block[] without HTML, for the same loop.

git targets src/git.ts

↑ back to cards

resolveScope is the only place that shells out. It maps each Target kind to a base/head pair: a commit diffs against its ^ parent, a branch against an explicit --base or the merge-base with main/master (defaultBase), the working tree against HEAD, and a PR is first gh pr checkout'd (the one hard dependency on the gh CLI) then treated as HEAD vs its merge-base. It returns a Scope — repoRoot, baseRef, headRef, a human label, and the raw unifiedDiff.

changedFiles is the structured companion: it merges git diff --numstat -z and --name-status -z into FileChange rows (path, A/M/D/R status, added/deleted counts). The NUL-delimited (-z) form is deliberate — it expands renames into explicit old/new tokens so a renamed file keeps its real new path and an R status. fileAtRef reads a file's contents at a ref (falling back to a working-tree read), used by the schema/api adapters downstream.

Key exports
resolveScope()→ Scope — base/head refs, label, and the raw unified diff.
changedFiles()→ FileChange[] — path, status, +/- counts, rename-aware.
fileAtRef()File contents at a ref (or working tree), "" if absent.
Target · ScopeThe target union and the resolved scope it produces.
Connections
spawnsgit for diffs; gh pr checkout for PR targets
feedsgather the Scope (raw diff) and changedFiles

diff parsing src/parse-diff.ts · src/diff-order.ts

↑ back to cards

parseUnifiedDiff is a small, pure state machine over the raw diff text: a diff --git line opens a new DiffBlock (title = basename, path = the b/ path), an @@ line opens a DiffHunk, and +/-/ lines append to the current hunk (the +++/--- file headers are skipped). No git, no I/O — just string work, which keeps it trivially testable.

sortByImportance is a stable rank sort so the reader meets the substance first: source code ranks 0, schema/config 2, styles 3, tests 4, and lockfiles/generated files 5 — ties preserve input order. buildBlocks runs both together (sortByImportance(parseUnifiedDiff(...))) so the diffs land in the block array already ordered.

Key exports
parseUnifiedDiff()Raw diff → one DiffBlock (with DiffHunks) per file.
sortByImportance()Stable rank sort: source → config → styles → tests → generated.
Connections
consumesscope.unifiedDiff from git targets
called bybuildBlocks — parse then sort the diffs

gather src/gather-recap.ts · src/recap-summary.ts

↑ back to cards

gatherRecap is the top of the pipeline: resolve the scope, get the changed files, selectAdapter (currently the Prisma/tRPC adapter, else a no-op default), then hand off to buildBlocks. It returns { scope, blocks, adapter } for the CLI to render.

buildBlocks composes the ordered Block[] that is the recap. It pushes a file-tree, asks the adapter for a schema diff and an API diff (each guarded so a failure only warns), unshifts a mechanical summaryMarkdown lead to the front, appends the code-intel dependencyNeighborhood "where it fits" diagram and the api-surface diagram, then the ranked diffs last. summaryMarkdown writes that lead — file/line totals, areas touched, and added/removed/changed procedures — as pure Markdown from the already-gathered data. The result is raw material: the visual-recap skill enriches it (overviews, annotations, groups) before the final render.

Key exports
gatherRecap()Target → { scope, blocks, adapter } — the whole pipeline.
buildBlocks()Compose the ordered Block[]: summary, files, diagrams, diffs.
summaryMarkdown()The mechanical Markdown lead (totals, areas, procedure deltas).

review shell src/assemble-review.ts · src/review/*.ts

↑ back to cards

assembleReview(blocks, opts) is the app-shell: it asserts unique ids, normalizes loose top-level diffs into walkthrough chapters (groupLooseDiffs), inlines review.css + the viewer script, renders every diagram once, then walks the blocks emitting <section>s — TL;DR + overview, files-changed, the guided walkthrough (progress rail + chapters), standalone diagram/schema/api/prose sections — into one self-contained HTML document. Along the way it runs lintBlocks and lintCompleteness, surfacing warnings (including any diagram that failed to compile) through onWarn.

The src/review/* renderers are the shell's vocabulary, each a focused function: topbar (title + risk/stat chips), sidebar (files + walkthrough outline + meta + progress rail), tldr (the What/Why/Risk/Size card + overview points), files-table, walkthrough (chapters and per-file subsections with diff bodies), diff (the raw +/- body), sections (diagram cards, the rendered legend, api-surface cards, reused-block fallbacks), diagrams (collect + render all), and normalize (stripChapterOrdinal, groupLooseDiffs). This shell is the domain's biggest shared seam — assemble-atlas and assemble-spec import these same renderers to draw the atlas and spec pages, so the topbar/sidebar/sections look and behave identically across all three products.

api-diagram src/api-diagram.ts

↑ back to cards

apiSurfaceDiagram turns a list of ApiProcedures (gathered by the stack adapter) into an architecture diagram of the changed API surface. It groups procedures by router, emits a d2 floor (client -> router, each procedure a node colored by its change — added/changed/removed), and a parallel mermaid upgrade with subgraph routers and the shared classDefs so the figure is editable in Excalidraw. It returns null when there are no procedures, so buildBlocks simply skips the section.

Key exports
apiSurfaceDiagram()→ DiagramBlock | null — routers→procedures, colored by change.
Connections
called bybuildBlocks with the adapter's gathered procedures
rendered bythe review shell via the shared diagrams renderer

completeness lint src/lint-completeness.ts

↑ back to cards

lintCompleteness is the diff-size-aware "demo standard" check — warnings only, it never throws or blocks rendering. If a recap has no diffs it is not a code recap and nothing is enforced. Otherwise it checks three things: a lead overview block exists with complete What/Why/Size facets and a risk level; every non-trivial diff (more than TRIVIAL_DIFF_LINES changed lines) carries a description; and once there are GROUP_THRESHOLD+ top-level diffs they are wrapped into group chapters rather than left as a flat pile. assembleReview runs it over the original blocks and routes its messages through onWarn, nudging the visual-recap skill to enrich the gather output past raw material.

Key exports
lintCompleteness()→ string[] of completeness warnings, in document order.
Connections
called byassembleReview — alongside lintBlocks, surfaced via onWarn
inspectsthe core OverviewBlock / DiffBlock shapes

Data it owns

recap owns the review app-shell vocabulary and the diff model; it borrows the generic Block union from core.

Review app-shellThe standing page vocabulary — topbar, sidebar, TL;DR, files-table, sections, walkthrough (chapters + subsections + progress rail) — defined by assemble-review.ts + src/review/* and reused by atlas + authoring.
FileChangeA changed file: path, status (A/M/D/R), and added/deleted line counts — rename-aware via git diff -z.
DiffBlock · DiffHunkThe parsed diff model: one block per file (title, path, hunks), each hunk an @@ header plus its +/-/ lines.
Target · ScopeThe git input union (pr / commit / branch / working) and the resolved scope (base/head refs, label, raw diff).
ReviewOpts · ReviewStatusThe render options (title, source, status, outDir) and the topbar status pill (green/yellow/red).

Reads (never owns): the generic Block union, OverviewBlock, ApiProcedure, and DiagramBlock from src/blocks.ts (core).
