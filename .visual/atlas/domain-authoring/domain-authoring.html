Domain

The two pure-authoring products — render a hand-written spec or doc straight into self-contained HTML, with no codebase scan.

Pathbin/spec.ts · bin/doc.ts · src/spec-blocks.ts · src/assemble-spec.ts · src/lint-spec.ts

RoleUnlike recap / atlas, the input is hand-authored blocks JSON, not a scan. spec renders an approval-focused design spec; doc is the general illustrated-doc renderer.
The load-bearing idea
"Two thin CLIs over the shared engine."

doc reuses core's generic Block model directly — bin/doc.ts just promotes mermaid fences and calls assemble, with almost no domain code of its own. spec adds its own approval-focused block vocabulary (SpecBlock — tldr, decisions, scope, rollout, approve) plus a dedicated assembleSpec renderer and a completeness lintSpec floor. Both ride the same diagram pipeline.

Two entrypoints, one core

architecture

spec.json flows through bin/spec.ts → assembleSpec (using spec-blocks + lintSpec); blocks.json flows through bin/doc.ts → core's assemble. Both share the diagrams pipeline (promote-mermaid for doc, renderAll for both) and the core renderers.

spec CLI bin/spec.ts

↑ back to cards

bin/spec.ts is the visual-spec entrypoint. It parseArgs for --blocks <spec.json> and --out <dir> (both required; relative paths resolve against the cwd), reads the JSON as a SpecDoc (page options + a blocks array), maps the page-level fields onto SpecOpts, and calls assembleSpec. It writes <out>/spec.html and re-writes <out>/spec.json, so the doc folder stays self-contained and re-renders in place — mirroring recap --blocks. Author warnings collected via onWarn are printed after the write.

doc CLI bin/doc.ts

↑ back to cards

bin/doc.ts is the thinnest product in the toolkit. It reads --blocks as core's generic Block[] (no spec model involved), runs promoteMermaidFences to lift mermaid code fences into real diagram blocks, then hands the result to core's assemble. It writes doc.html plus a copy of blocks.json into the --out directory so the doc is self-contained and re-renders in place. There is no assemble-doc, no doc-blocks, and no lint — doc is wiring over core + diagrams.

spec block model src/spec-blocks.ts

↑ back to cards

The SpecBlock discriminated union — the approval-focused vocabulary that distinguishes a spec from a generic doc: tldr (the lead), diagram (info-flow hero / where-it-fits), components (with the optional two-layer anatomy explainer), fits, decisions (the highest-value content for an approver), scope (in/out boundaries), rollout (gated phases), done, risks, approve (the commit / scrutinize / open-questions band), reference (drill-down drawers), and the spec-prose escape hatch. Helpers derive the page structure: isChapter (tldr + reference are not numbered chapters), assertUniqueSpecIds, chapterLabel, and collectSpecDiagrams / toDiagramBlock to feed the shared diagram pipeline.

spec render src/assemble-spec.ts

↑ back to cards

assembleSpec(blocks, opts) is the spec page builder. It asserts unique ids, loads review.css + spec.css + the shared viewer JS + the dark-mode toggle assets (theme-head.js, theme-toggle.js, theme.css), collects the spec diagrams and renders them via the diagrams domain's renderAll, then runs the page through validateSpecOpts and lintSpec (warnings only). Each block becomes a <section> via a per-type renderer; the sidebar outline and the progress rail are derived from the chapters (the rail is injected right after the TL;DR). For diagram blocks it reuses the recap's renderDiagramCard — the one piece of review-shell rendering authoring borrows.

spec lint src/lint-spec.ts

↑ back to cards

lintSpec(blocks) is the demo-standard completeness floor — a mechanical backstop that keeps every spec at the bar. It returns warning strings (never throws): no tldr lead, an empty TL;DR, missing decisions (the load-bearing content for an approver) or a decision lacking a why, and a missing scope block. For a large spec (≥ 5 chapters) it also expects a TL;DR big-idea line, a hero diagram, a rollout, and an approve band; independently of size, four-plus decisions with no named rejected-alternative earn a nudge to name the path not taken on the contested ones. assembleSpec surfaces these through onWarn.

Key files
lint-spec.tsThe heuristic checks (lead / decisions / scope, scaled stricter at LARGE_CHAPTERS = 5).
Key exports

Connections

Data it owns

Authoring owns the spec block model — the approval-focused vocabulary in spec-blocks.ts. doc owns no model of its own; it borrows core's generic Block.

TldrBlocktype: "tldr" — the lead: a one-line heading, What / Why / Closes / Size rows, and an optional big-idea line.
SpecDiagramBlocktype: "diagram" — a standalone d2/mermaid section (info-flow hero, where-it-fits); its id is both section and diagram id.
ComponentsBlock · ComponentCardtype: "components" — the component grid, with the optional two-layer anatomy (factual / perceived) explainer.
FitsBlocktype: "fits" — the where-it-fits chain plus an optional new-vs-reused layer stack.
DecisionsBlock · DecisionItemtype: "decisions" — Q / A / why / rejected cards; the highest-value content for an approver.
ScopeBlocktype: "scope" — in-scope goals vs out-of-scope anti-goals, with optional defer targets.
RolloutBlock · PhaseItemtype: "rollout" — gated sub-phases, each with a scope and an acceptance gate.
DoneBlock · RisksBlocktype: "done" (movers + before/after table) and type: "risks" (risk / mitigation cards).
ApproveBlocktype: "approve" — the sign-off band: what you're approving, what to scrutinize, and open non-blocking questions.
ReferenceBlock · SpecProseBlocktype: "reference" (collapsed drill-down drawers) and type: "spec-prose" (the unmodeled-content escape hatch).

All of these compose into SpecBlock. Borrows (never owns): DiagramBlock / DiagramKind / TabsBlock from core; the generic Block model that doc renders.
